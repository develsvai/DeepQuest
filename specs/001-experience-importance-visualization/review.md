# Post-Implementation Review: Spec Design Lessons

**Review Date**: 2025-10-14
**Purpose**: Document universal lessons for future spec creation

---

## What Went Wrong

### Problem: Spec과 실제 구현의 심각한 괴리

**예상**: ~7개 파일 수정
**실제**: 17개 파일 수정 (143% 증가)

Plan.md의 "Project Structure"에서 예측한 파일들과 실제 `git status`의 변경사항이 완전히 달랐다.

---

## Root Cause: 데이터 흐름 추적 실패

### 놓친 것

Spec은 **UI와 DB만** 보고 작성되었다:
- ✅ Prisma 스키마 수정
- ✅ React 컴포넌트 수정
- ❌ **데이터가 어떻게 생성되고 전달되는지 전혀 파악하지 않음**

### 실제 필요했던 것

데이터는 여러 레이어를 거쳐 흐른다:
1. **생성**: AI 서버에서 데이터 생성
2. **전달**: 웹훅/API를 통해 프론트엔드로 전송
3. **변환**: 중간 레이어에서 타입 변환
4. **저장**: 서비스 레이어에서 DB에 저장
5. **조회**: API에서 DB 읽기
6. **표시**: UI 컴포넌트에서 렌더링

Spec은 4, 6번만 보고 1~3, 5번을 완전히 무시했다.

---

## 핵심 교훈

### 1. **"거꾸로 추적" 필수**

UI에서 시작해서 데이터 소스까지 역방향으로 추적하라:

```
[UI 컴포넌트]
  ↓ 어디서 데이터를 받나?
[API/tRPC]
  ↓ 어디서 데이터를 읽나?
[Database]
  ↓ 누가 저장하나?
[Service Layer]
  ↓ 어디서 데이터를 받나?
[Webhook/API Handler]
  ↓ 누가 보내나?
[Data Source (AI/External API/etc)]
```

**각 단계마다 구체적인 파일 경로를 적어라.**

---

### 2. **코드를 읽지 않고 추측하지 마라**

Plan.md에 "✅ REUSE: No changes"라고 표시한 파일들이 실제로는 수정되었다.

**잘못된 접근**:
- 문서만 읽고 "아마도 이럴 것 같다" 추측
- 파일 이름만 보고 "변경 불필요"라고 판단

---

### 3. **중간 레이어 놓치지 마라**

데이터가 A에서 B로 바로 가지 않는다. 중간에 여러 레이어가 있다:

- **타입 변환 레이어**: Pydantic → Zod → Prisma → React Props
- **데이터 전달 레이어**: 웹훅 핸들러, API 라우터
- **비즈니스 로직 레이어**: 서비스, 유틸리티 함수

각 레이어마다 파일 수정이 필요할 수 있다.

## 다음 Spec 작성 시 필수 체크리스트

### Phase 0: Research (Spec 작성 전)

```markdown
## Pre-Spec Investigation Checklist

### □ 데이터 소스 파악
- [ ] 데이터가 어디서 생성/시작되는가? (파일 경로 기록)
- [ ] 해당 코드를 실제로 읽었는가?

### □ 데이터 흐름 추적
- [ ] 데이터가 거치는 모든 레이어 나열 (파일 경로 포함)
- [ ] 각 레이어에서 타입 변환이 발생하는가?
- [ ] 중간 변환/검증 로직이 있는가?

### □ 데이터 저장 확인
- [ ] 어떤 서비스/함수가 데이터를 저장하는가?
- [ ] DB 스키마 변경이 필요한가?
- [ ] 마이그레이션 전략은?

### □ 데이터 소비 확인
- [ ] 어떤 API/쿼리가 데이터를 제공하는가?
- [ ] UI 컴포넌트는 어떻게 데이터를 받는가?

### □ 파일 목록 검증
- [ ] 예상 변경 파일 목록 작성 완료
- [ ] 각 파일의 변경 이유 명확히 기술
- [ ] "추측"이 아닌 "확인된 사실" 기반
```

---

## 체크리스트 사용법

### Spec 작성 전

1. 위 체크리스트를 `research.md`에 복사
2. 모든 항목을 실제로 수행 (명령어 실행, 파일 읽기)
3. 각 항목의 결과를 문서화
4. **모든 체크박스가 체크되기 전까지 spec.md/plan.md 작성 금지**

### Plan.md 작성 시

"Project Structure" 섹션에서:
- "추측"으로 파일 목록 작성 금지
- 실제 코드 분석 기반으로만 작성
- 각 파일의 변경 이유를 구체적으로 기술

### Spec 검증 시

다음 질문에 답할 수 없으면 spec이 불완전한 것:
- 데이터가 정확히 어디서 생성되는가? (파일:라인)
- 데이터가 거치는 모든 중간 레이어는? (파일 경로 나열)

---

## 핵심 원칙

> **"추적하지 않은 것은 spec으로 작성하지 마라"**

UI에서 데이터 소스까지 전체 경로를 파일 레벨로 추적하지 못했다면, 그 spec은 아직 작성할 준비가 안 된 것이다.

## 다음 Spec에서 반드시 포함해야 할 섹션

### research.md에 추가

```markdown
## Data Flow Tracing

### End-to-End Path
[UI Component] → [API] → [Service] → [Data Source]

파일 경로:
1. [UI]: `경로/파일명.tsx`
2. [API]: `경로/파일명.ts`
3. [Service]: `경로/파일명.ts`
4. [Source]: `경로/파일명.ts`

### Type Transformations
각 경계에서 타입이 어떻게 변환되는지 명시
```

## 이 Review의 핵심 메시지

**문제**: Spec이 "UI + DB"만 보고 중간 데이터 파이프라인을 완전히 무시함

**원인**:
- 코드를 읽지 않고 추측으로 작성
- 데이터 흐름을 추적하지 않음

**해결**:
- ✅ 거꾸로 추적 (UI → 데이터 소스)
- ✅ 실제 코드 읽기 (추측 금지)
- ✅ 모든 중간 레이어 문서화

**다음 Spec에 적용**:
- Pre-Spec Investigation Checklist 필수 수행
- research.md에 Data Flow Tracing 섹션 필수 작성
- Plan.md의 파일 목록은 실제 분석 기반으로만 작성
