## 1. Database / PostgreSQL

### 1.1 initdb 실패

**에러:** `initdb: error: directory "/var/lib/postgresql/data" exists but is not empty`

**원인:** 기존 Docker 볼륨에 데이터가 남아 있어 PostgreSQL 초기화가 실패합니다.

**조치:** 볼륨까지 삭제 후 재시작합니다.

```bash
# PostgreSQL compose 파일이 있는 디렉터리로 이동
cd infra/docker/postgres
# 볼륨까지 포함하여 컨테이너와 볼륨 모두 삭제
docker-compose -f docker-compose.yml down -v
```

---

### 1.2 볼륨 마운트 충돌

**문제:** `logs` 디렉터리를 PostgreSQL data 디렉터리 내부에 마운트하면 initdb와 충돌합니다.

**원인:** PostgreSQL이 초기화할 때 data 디렉터리가 비어있어야 하는데, logs 디렉터리가 이미 존재하면 충돌이 발생합니다.

**조치:**

- data 디렉터리 내부 마운트 제거
- 로그 디렉터리는 별도 경로 사용 (예: `/var/log/postgresql` 또는 상위 디렉터리)

---

### 1.3 K8s Pod CrashLoopBackOff (권한 에러)

**에러:** `chmod: /var/run/postgresql: Operation not permitted`, `Permission denied: '/etc/postgresql/postgresql.conf'`

**원인:** K8s StatefulSet에서 PostgreSQL이 필요한 디렉터리/설정 파일에 대한 권한이 없거나, 볼륨 마운트가 초기화와 충돌합니다.

**조치:**

1. StatefulSet에 `fsGroup: 999` 추가 (PostgreSQL 실행 유저와 GID 일치).
2. 컨테이너의 `securityContext` 제거 (권한 충돌 방지).
3. `postgresql.conf` 볼륨 마운트 제거 후 기본 설정 사용.

```yaml
# statefulset.yaml 예시 (spec.template.spec)
spec:
  securityContext:
    fsGroup: 999   # postgres 그룹 GID
  containers:
    - name: postgres
      # securityContext 제거
      # postgresql.conf volumeMount 제거
```

---

### 1.4 데이터베이스 없음 (K8s)

**에러:** `FATAL: database 'deepquest' does not exist`

**원인:** PostgreSQL Pod는 기동되었지만 애플리케이션용 데이터베이스가 생성되지 않았습니다. `POSTGRES_DB`만으로는 생성이 안 되는 구성일 수 있습니다.

**조치:** Postgres Pod에서 수동으로 DB를 생성합니다.

```bash
# deep-quest 네임스페이스, postgres-0 Pod 기준
kubectl exec -n deep-quest postgres-0 -- psql -U deepquest -d postgres -c 'CREATE DATABASE deepquest;'
```

---

### 1.5 pg_hba.conf 원격 접속 불허 (K8s)

**에러:** `User was denied access on the database deepquest`

**원인:** 기본 `pg_hba.conf`가 클러스터 내부/원격 접속을 허용하지 않습니다.

**조치:** `pg_hba.conf`에 원격 접속 규칙을 추가한 뒤 PostgreSQL을 리로드합니다.

```bash
kubectl exec -n deep-quest postgres-0 -- su postgres -c \
  'echo "host all all 0.0.0.0/0 md5" >> /var/lib/postgresql/data/pg_hba.conf && pg_ctl reload -D /var/lib/postgresql/data'
```

## 2. Prisma / DB 연결

### 2.1 DIRECT_URL 누락

**에러:** `Environment variable not found: DIRECT_URL`

**원인:** `web/.env.local` 또는 Docker env 파일에 `DIRECT_URL`이 누락되었습니다. Prisma는 마이그레이션 실행 시 `DIRECT_URL`을 사용합니다.

**조치:** 환경 변수 파일에 `DIRECT_URL`을 추가합니다.

```bash
# 로컬 개발 환경 (web/.env.local)
DATABASE_URL=postgresql://user:password@host:5432/dbname  # Prisma Client용 연결 URL
DIRECT_URL=postgresql://user:password@host:5432/dbname  # 마이그레이션 실행용 직접 연결 URL
NODE_ENV=development
```

**Docker Compose 사용 시:** `infra/docker/web/.env`(또는 `.env.dev`, `.env.prod`) 파일에 `DIRECT_URL` 포함 여부를 확인합니다.

---

### 2.2 DB 인증 실패 (P1000)

**에러:** `P1000: Authentication failed`

**주요 원인:**

- Web / Postgres 비밀번호 불일치
- 비밀번호에 `@` 포함 시 URL 인코딩 필요 (`@` → `%40`)

**조치:**

1. 웹과 Postgres에서 사용하는 비밀번호를 동일하게 맞춥니다.
2. 비밀번호에 `@`가 있으면 URL 인코딩을 적용합니다.

```bash
# infra/docker/web/.env 예시 (비밀번호에 @ 없음)
DATABASE_URL=postgresql://deepquest:smt32f734o@postgres-host:5432/deepquest
DIRECT_URL=postgresql://deepquest:smt32f734o@postgres-host:5432/deepquest

# 비밀번호에 @가 있는 경우 (@ → %40)
DATABASE_URL=postgresql://user:pass%40word@host:5432/db
DIRECT_URL=postgresql://user:pass%40word@host:5432/db
```

**K8s 환경:** `apply-secrets.sh`에서 `@` → `%40` 인코딩이 자동 적용됩니다. 자세한 내용은 [K8S_배포 가이드](./K8S_배포%20가이드.md)를 참고하세요.

---

### 2.3 테이블 없음 (P2021)

**에러:** `P2021: Table does not exist`

**원인:** Prisma 마이그레이션이 실행되지 않아 데이터베이스 테이블이 생성되지 않았습니다.

**조치:** 로컬 환경에서 마이그레이션을 실행합니다.

```bash
# web 디렉터리로 이동
cd web
# 환경 변수 설정 후 마이그레이션 실행
DATABASE_URL='postgresql://user:password@localhost:5432/dbname' \
DIRECT_URL='postgresql://user:password@localhost:5432/dbname' \
npx prisma migrate deploy
```

**Docker Compose 환경:** 웹 컨테이너에서 마이그레이션을 실행합니다.

```bash
# 컨테이너 이름은 사용 중인 compose 파일의 서비스 이름에 맞게 조정
docker exec deepquest-web sh -c "cd /app && npx prisma migrate deploy"
# infra/docker 통합 compose면 서비스 이름(예: web-server)에 맞게 실행
```

**K8s 환경:** 클러스터 내 Postgres에 직접 접근할 수 없으므로 port-forward 후 로컬에서 마이그레이션을 실행합니다.

```bash
# 터미널 1: Postgres Service 포트 포워드 (로컬 5433 → 서비스 5432)
kubectl port-forward -n deep-quest svc/postgres-service 5433:5432

# 터미널 2: web 디렉터리에서 연결 정보를 로컬 포트에 맞춰 마이그레이션
cd web
DATABASE_URL='postgresql://deepquest:비밀번호@localhost:5433/deepquest' \
DIRECT_URL='postgresql://deepquest:비밀번호@localhost:5433/deepquest' \
npx prisma migrate deploy
```

### 2.4 tRPC An unexpected error occurred

**증상:** 로그인은 되는데 대시보드/데이터 조회 시 `An unexpected error occurred` 메시지가 표시됩니다.

**원인:**

- DB 인증 실패: `DATABASE_URL` 또는 비밀번호 불일치로 데이터베이스 연결 실패
- Prisma pg adapter 번들링: Next.js가 Prisma의 PostgreSQL adapter를 번들링하지 않아 런타임 오류 발생
- LangGraph 404: thread 소실로 인한 일시적 장애

**조치:**

1. DB 통합: `apply-secrets.sh`로 Secret 재적용하여 DB 연결 정보 통일
2. Next.js externalPackages 설정: `next.config.ts`에 Prisma 관련 패키지를 externalPackages로 추가
3. LangGraph 404: 일시 장애로 판단, 재시도 후 정상화

**Next.js 설정:**

```typescript
// next.config.ts
const nextConfig = {
  experimental: {
    // Prisma PostgreSQL adapter를 번들링하지 않도록 설정
    serverComponentsExternalPackages: ['@prisma/client', 'pg'],
  },
};
```

### 2.5 K8s DB 권한 부족

**에러:** `User was denied access on the database deepquest` (연결은 되나 테이블/스키마 접근 불가)

**원인:** DB 사용자에게 `public` 스키마 및 테이블·시퀀스에 대한 권한이 부여되지 않았습니다.

**조치:** Postgres Pod에서 해당 사용자에게 권한을 부여합니다.

```bash
kubectl exec -n deep-quest postgres-0 -- psql -U deepquest -d deepquest -c \
  'GRANT ALL PRIVILEGES ON ALL TABLES IN SCHEMA public TO deepquest;
   GRANT ALL PRIVILEGES ON ALL SEQUENCES IN SCHEMA public TO deepquest;
   GRANT ALL PRIVILEGES ON SCHEMA public TO deepquest;'
```

## 3. AI 서버 (Python / uv / LangGraph)

### 3.1 uv 명령어 없음

**에러:** `uv: not found`

**원인:** `uv`가 설치되었지만 PATH에 포함되지 않았습니다. `uv`는 `~/.local/bin`에 설치되므로 PATH에 추가해야 합니다.

**조치:** Dockerfile에 PATH 환경 변수를 추가합니다.

```dockerfile
# uv 설치 후 다음 줄에 추가
ENV PATH="/root/.local/bin:$PATH"
```

---

### 3.2 모듈 import 실패

**에러:** `ModuleNotFoundError: No module named 'jd_to_text'`

**원인:** AI 프로젝트의 로컬 패키지가 editable 모드로 설치되지 않았습니다. 소스 코드를 복사한 후 editable 설치가 필요합니다.

**조치:** `infra/docker/ai/Dockerfile`에서 소스 복사 후 editable 설치를 실행합니다.

```dockerfile
# 소스 코드 복사
COPY . /app

# editable 모드로 설치 (현재 디렉터리의 패키지를 시스템에 설치)
RUN uv pip install --system -e .
```

---

### 3.3 Healthcheck 실패 / unhealthy

**증상:** 컨테이너가 unhealthy 상태로 표시됩니다.

**원인:** LangGraph의 기본 헬스체크 경로는 `/ok`입니다. compose/Dockerfile의 healthcheck 경로가 일치하지 않으면 실패합니다.

**조치:** 헬스체크 경로를 통일합니다.

```yaml
# docker-compose.yml 예시
healthcheck:
  test: ["CMD", "curl", "-f", "http://localhost:8123/ok"]  # LangGraph 기본 헬스 경로
  interval: 30s
  timeout: 10s
  retries: 3
```

```dockerfile
# Dockerfile 예시
HEALTHCHECK --interval=30s --timeout=10s --retries=3 \
  CMD curl -f http://localhost:8123/ok || exit 1
```

---

### 3.4 LangGraph 캐시 디렉토리 권한 (K8s)

**에러:** `PermissionError: [Errno 13] Permission denied: '.langgraph_api'`

**원인:** LangGraph API가 캐시용으로 사용하는 `.langgraph_api` 디렉터리를 컨테이너 내에서 생성·쓰기할 권한이 없습니다. K8s 환경에서는 쓰기 가능한 볼륨이 필요합니다.

**조치:** AI Deployment에 `emptyDir` 볼륨을 마운트하고, 워킹 디렉터리를 `/app`으로 고정합니다.

```yaml
# deployment.yaml (AI) - containers 항목
containers:
  - name: ai-server
    workingDir: /app
    volumeMounts:
      - name: langgraph-cache
        mountPath: /app/.langgraph_api
# volumes 항목
volumes:
  - name: langgraph-cache
    emptyDir: {}
```

## 4. Web 빌드 & 런타임

### 4.1 pnpm lockfile 오류

**에러:** `ERR_PNPM_NO_LOCKFILE`

**원인:** 빌드 컨텍스트에 `pnpm-lock.yaml`이 없는데 `--frozen-lockfile` 옵션을 사용하면 실패합니다.

**조치:**

1. `--frozen-lockfile` 옵션 제거
2. 또는 빌드 컨텍스트에 `pnpm-lock.yaml` 포함

```dockerfile
# infra/docker/web/Dockerfile
# lockfile이 컨텍스트에 포함된 경우
COPY pnpm-lock.yaml ./
RUN pnpm install --frozen-lockfile

# lockfile이 없는 경우 (권장하지 않음)
RUN pnpm install
```

---

### 4.2 Prisma 디렉터리 COPY 오류

**에러:** `/app/node_modules/.prisma not found`

**원인:** Prisma Client는 빌드 단계에서 `prisma generate` 실행 시 생성됩니다. 런타임에 별도로 COPY할 필요가 없습니다.

**조치:** Dockerfile에서 `.prisma`/`@prisma` 디렉터리를 COPY하는 라인을 제거합니다.

```dockerfile
# 잘못된 예시 (제거해야 함)
COPY node_modules/.prisma ./node_modules/.prisma
COPY node_modules/@prisma ./node_modules/@prisma

# 올바른 방법: prisma generate는 빌드 단계에서 실행되므로 별도 COPY 불필요
RUN npx prisma generate
# 이후 node_modules 전체가 COPY되거나 이미 포함되어 있으면 됨
```

## 5. 인증 / Clerk 관련

### 5.1 로그인 후 404

**증상:** 로그인 후 `/ko/auth/signin` 등으로 리디렉션되면 404 에러가 발생합니다.

**원인:** i18n을 사용하는 경우 리디렉션 URL에 locale prefix(`/ko`)가 필요합니다. Clerk의 기본 리디렉션 URL에 locale이 없으면 404가 발생합니다.

**조치:** 환경 변수에 locale prefix를 포함한 리디렉션 URL을 설정합니다.

```bash
# infra/docker/web/.env 또는 동일 용도 파일
NEXT_PUBLIC_CLERK_SIGN_IN_FALLBACK_REDIRECT_URL=/ko/dashboard  # 로그인 후 이동할 URL
NEXT_PUBLIC_CLERK_SIGN_UP_FALLBACK_REDIRECT_URL=/ko/dashboard  # 회원가입 후 이동할 URL
```

---

### 5.2 설정 변경 미반영

**증상:** 환경 변수를 변경했는데 브라우저에서 변경사항이 반영되지 않습니다.

**원인:** 브라우저 캐시 또는 Clerk SDK의 로컬 스토리지에 이전 설정이 남아있습니다.

**조치:**

1. 브라우저 캐시 삭제
2. Clerk SDK 로컬 스토리지 삭제 (개발자 도구 → Application → Local Storage)
3. 시크릿 모드로 재접속하여 테스트

---

### 5.3 Clerk Publishable Key 에러 (K8s)

**에러:** `Error: Publishable key not valid.`

**원인:** Web Pod에 주입된 `NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY`가 비어 있거나, test/live 혼용·오타 등으로 유효하지 않습니다. K8s에서는 `web-secret`(또는 동일 용도 Secret)에 실제 값이 들어 있어야 합니다.

**조치:** `web-secret`을 실제 Clerk Publishable Key(및 필요한 나머지 값)로 재생성합니다.

```bash
kubectl create secret generic web-secret -n deep-quest \
  --from-literal=NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY='pk_test_...' \
  --from-literal=CLERK_SECRET_KEY='sk_test_...' \
  # ... 기타 필수 변수 (DATABASE_URL, CLERK_WEBHOOK_SIGNING_SECRET 등)
  --dry-run=client -o yaml | kubectl apply -f -
```

프로덕션에서는 `pk_live_*` / `sk_live_*`를 사용해야 합니다. 전체 시크릿 예시는 [10. K8s 초기 설정 / 시크릿](#10-k8s-초기-설정--시크릿)을 참고하세요.

## 6. Kubernetes / Prod 이슈

### 6.1 Immutable 필드 오류

**문제:** ArgoCD/Kustomize 적용 시 Deployment/StatefulSet의 selector 불일치로 업데이트가 실패합니다.

**원인:** Kubernetes에서 Deployment/StatefulSet의 selector는 immutable 필드입니다. 기존 리소스의 selector를 변경하려고 하면 오류가 발생합니다.

**조치:**

1. `kustomization.yaml`에 `includeSelectors: false` 적용
2. StatefulSet PVC 관련 패치 제거

```yaml
# kustomization.yaml 예시
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization
resources:
  - ../../base
patches:
  - path: deployment-patch.yaml
    target:
      kind: Deployment
# selector 변경을 허용
includeSelectors: false
```

---

### 6.2 HPA 중복

**문제:** prod overlay에 HPA가 base와 중복으로 패치되어 충돌이 발생합니다.

**원인:** base에 이미 HPA가 정의되어 있는데 overlay에서 다시 정의하면 중복이 발생합니다.

**조치:**

- prod overlay의 `hpa.yaml` 제거 (이미 적용됨)
- base HPA 패치만 사용 (min 2, max 5)

**현재 설정:**

- Base HPA: AI min 1/max 3, Web min 1/max 5
- Prod 패치: AI/Web 모두 min 2, max 5로 오버라이드

---

### 6.3 Kustomize commonLabels deprecated

**경고:** `'commonLabels' is deprecated. Please use 'labels' instead.`

**원인:** Kustomize에서 `commonLabels` 필드가 deprecated되어 `labels` 사용을 권장합니다.

### 6.7 Service Endpoints 비어 있음 (Web/Postgres/AI 파드 안 뜸) — **dev overlay 한정**

**적용 환경:** 이 현상은 **dev overlay**로 배포했을 때만 발생합니다. prod overlay는 `includeSelectors: false`라서 Service selector가 바뀌지 않으며, 따라서 **production 배포에서는 이 문제가 나지 않습니다.**

**증상:** `kubectl get endpoints postgres-service` 또는 `web-service`, `ai-service`가 `ENDPOINTS <none>`으로 나오고, Web Pod init 컨테이너에서 "PostgreSQL is unavailable - sleeping" / "AI Server is unavailable" 로 멈춤.

**원인:** dev overlay의 `labels.pairs`에 `environment: dev`를 넣고 **`includeSelectors: true`**로 두면, Kustomize가 Service의 selector에 `environment: dev`를 붙입니다. 반면 base의 Deployment/StatefulSet pod template에는 해당 라벨이 없을 수 있어, 생성된 Pod가 Service selector와 매칭되지 않습니다.

**즉시 조치 (dev 배포 시):** 해당 Pod에 라벨을 붙여 Endpoints를 채웁니다.

```bash
# Postgres (StatefulSet 파드)
kubectl label pod postgres-0 -n deep-quest environment=dev --overwrite

# AI Deployment 파드
kubectl label pods -n deep-quest -l app.kubernetes.io/component=ai environment=dev --overwrite

# Web Deployment 파드
kubectl label pods -n deep-quest -l app.kubernetes.io/component=web environment=dev --overwrite
```

**영구 조치:** dev overlay에서 Deployment/StatefulSet의 pod template에 `environment: dev` 라벨이 들어가도록 패치를 추가하거나, dev에서 `includeSelectors: false`로 바꿔 Service selector를 건드리지 않게 합니다.

## 7. Tailscale Funnel

### 7.1 구조 변경 이유

**기존 구조:** web-server Pod에 Tailscale Funnel을 사이드카로 구성했습니다.

**변경된 구조:** Standalone Funnel Deployment로 분리했습니다.

**변경 이유:**

- Funnel은 localhost 기반으로 동작합니다. 사이드카로 구성하면 web-server Pod 내부에서만 접근 가능합니다.
- Clerk webhook 등 외부 서비스에서 HTTPS 공개 URL이 필요합니다. Funnel을 standalone으로 분리하여 외부에서 접근 가능한 URL을 제공합니다.

---

### 7.2 핵심 설정

**L4 프록시:** socat을 사용하여 `localhost:3000` → `web-service:80`으로 포워딩합니다.

```yaml
# tailscale-funnel-deployment.yaml 예시
containers:
  - name: socat
    image: alpine/socat
    args:
      - tcp-listen:3000,fork,reuseaddr  # Funnel이 접근하는 포트
      - tcp-connect:web-service:80      # 실제 web 서비스로 포워딩
```

**TS_KUBE_SECRET:** dev/prod 환경별로 분리된 Secret을 사용합니다.

- Prod: `tailscale-funnel-prod-state`
- Dev: `tailscale-funnel-dev-state`

**TS_HOSTNAME:** 단일화된 호스트명을 사용합니다.

- 변경 전: `web-server-funnel-prod`
- 변경 후: `deepquest`

**Funnel 정책 설정:**

1. Tailscale Admin 콘솔 접속: https://login.tailscale.com/f/funnel?node=xxx
2. 해당 노드에 Funnel 권한 부여 (노드가 이미 Tailnet에 연결되어 있어야 함)
3. Funnel URL: `https://deepquest.tail2dac17.ts.net`

---

### 7.3 Tailscale-auth / State Secret 구조 이해

**문제:** Tailscale state Secret이 존재하면 auth key 없이 재연결이 가능하다고 오해했습니다.

**원인:** Tailscale state Secret이 있어도 재연결 시 auth key가 필요할 수 있습니다. Funnel 권한은 인증 이후 별도로 설정해야 합니다.

**조치:**

- TS_AUTHKEY 방어적 추가: state Secret이 있어도 재연결 시 auth key가 필요할 수 있으므로 항상 포함합니다.
- Funnel 권한 별도 설정: Tailscale Admin 콘솔에서 노드에 Funnel 권한 부여

**현재 정책:**

- `tailscale-auth` Secret에 TS_AUTHKEY 항상 포함
- Funnel 권한은 Tailscale Admin → https://login.tailscale.com/f/funnel?node=xxx 에서 설정

**결과:**

- 복구 안정성 확보: auth key가 항상 포함되어 있어 재연결 시 문제 발생 가능성 감소

### 9.1 spawn helper 오류

**에러 메시지:**

```
Error when executing always post condition:
java.io.IOException: error=0, Failed to exec spawn helper: pid: XXXXX, exit value: 1
...
Caused: java.io.IOException: Cannot run program "nohup" (in directory "/home/jenkins-build/workspace/..."): error=0, Failed to exec spawn helper
```

**원인:** Jenkins 에이전트(`docker-build`)에서 `post { always { sh ... } }` 실행 시 셸 프로세스 스폰이 실패합니다.

**주요 원인:**

- 에이전트 리소스 부족 (메모리/프로세스 한도 초과)
- PATH 문제 (`nohup` 명령어 없음)
- 컨테이너/제한 환경에서 exec 제한

---

### 9.2 대응 전략

**1. 에이전트 재시작**

`docker-build` 라벨 에이전트 노드를 재시작한 후 파이프라인을 재실행합니다.

**2. 리소스 확인**

에이전트 호스트에서 메모리/프로세스 한도를 확인합니다.

```bash
# 프로세스 한도 확인
ulimit -u

# 메모리 오버커밋 설정 확인
cat /proc/sys/vm/overcommit_memory
```

**3. Jenkinsfile 완화 (적용됨)**

- `catchError`로 감싸서 실패해도 파이프라인 결과에 영향 없음
- `BUILD_TAG`가 있을 때만 `docker rmi` 실행
- 단일 `sh` 실행으로 스폰 횟수 감소

---

### 9.3 현재 적용 정책

**현재 적용된 코드:**

```groovy
post {
    always {
        script {
            // catchError로 감싸서 실패해도 파이프라인 결과에 영향 없음
            catchError(buildResult: null, stageResult: null) {
                // BUILD_TAG가 있을 때만 이미지 삭제 실행
                if (env.BUILD_TAG?.trim()) {
                    // 한 줄로 통합하여 스폰 횟수 감소
                    sh "docker rmi ${HARBOR_REGISTRY}/${HARBOR_PROJECT}/ai:${env.BUILD_TAG} 2>/dev/null || true; docker rmi ${HARBOR_REGISTRY}/${HARBOR_PROJECT}/web:${env.BUILD_TAG} 2>/dev/null || true"
                }
            }
        }
        // 작업 디렉터리 정리
        deleteDir()
    }
}
```

## 11. AI 파드 ImagePullBackOff (Harbor 이미지 not found)

**에러 메시지:**

```
Failed to pull image "harbor.192.168.0.110.nip.io/deep-quest/ai:140-65141ab": not found
Error: ImagePullBackOff
```

**원인:** ArgoCD/deploy 브랜치가 가리키는 이미지 태그(예: `140-65141ab`)가 Harbor에 없음. Jenkins에서 해당 빌드 시 AI 이미지를 푸시하지 않았거나, 빌드 실패 후 deploy 브랜치만 갱신된 경우.

**대응 (둘 중 하나)**

1. **Jenkins로 해당 태그 푸시하기**

   - 빌드가 성공하면 `ai:${BUILD_TAG}` 와 `ai:prod` 가 Harbor에 푸시되고, deploy 브랜치가 새 BUILD_TAG로 갱신됨.  
   - ArgoCD Sync 후 파드가 새 이미지를 pull.

2. **deploy 브랜치 태그를 Harbor에 있는 이미지로 맞추기**  
   - Harbor에 실제로 있는 AI 이미지 태그 확인 (예: `prod`, `59-ffb2ec8`).  
   - 인프라 레포(deep-quest-infra) **deploy** 브랜치에서 `k8s/overlays/prod/kustomization.yaml` 의 ai-tag를 해당 태그로 수정 후 push.  
   - ArgoCD Sync 하면 해당 이미지로 배포됨.

## 12. AI 파드 크래시 — License verification failed

**에러 로그:**

```
ValueError: License verification failed. Please ensure proper configuration:
- For local development, set a valid LANGSMITH_API_KEY for an account with LangGraph Cloud access
- For production, configure the LANGGRAPH_CLOUD_LICENSE_KEY environment variable
Application startup failed. Exiting.
```

**원인:** `langgraph build` 이미지(langgraph-api 0.7.x licensed)는 기동 시 LangSmith/License 검증을 하며, 실패하면 **예외를 던져 프로세스가 종료**됩니다. ai-secret의 `LANGSMITH_API_KEY`가 비어 있거나, 플레이스홀더이거나, 해당 계정에 LangGraph Cloud(self-hosted lite) 권한이 없으면 403 후 위 에러가 납니다.

**대응 (둘 중 하나 필수):**

1. **LANGSMITH_API_KEY 사용 (개발/무료 Self-hosted lite)**  
   - [LangSmith](https://smith.langchain.com) 가입 후 API Key 발급.  
   - **LangGraph Cloud(self-hosted lite)** 가 활성화된 계정이어야 합니다.  
   - ai-secret에 실제 키 반영: `infra/k8s/scripts/apply-secrets.sh` 실행 전 `docker/ai/.env.prod`(또는 사용하는 .env)에 `LANGSMITH_API_KEY=lsv2_pt_...` 설정 후 apply.

2. **LANGGRAPH_CLOUD_LICENSE_KEY 사용 (프로덕션 라이선스 보유 시)**  
   - ai-secret에 `LANGGRAPH_CLOUD_LICENSE_KEY` 키를 추가하고 값을 설정한 뒤, apply-secrets로 반영.  
   - Deployment는 이미 ai-secret을 envFrom으로 읽으므로, 시크릿만 갱신 후 파드 재시작하면 됨.

**시크릿 반영 후:**

```bash
kubectl rollout restart deployment ai-server -n deep-quest
```
