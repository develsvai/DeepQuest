# Ansible 자동화 후보 정리

현재 인프라(`infra/`)에서 수동/스크립트로 수행하는 작업 중 Ansible로 자동화할 수 있는 항목을 정리한 문서입니다.

---

## 1. 요약 표

| 영역 | 현재 방식 | Ansible 자동화 후보 | 우선순위 |
|------|-----------|---------------------|----------|
| **Secrets 적용** | `apply-secrets.sh` + 수동 Harbor/TLS | Playbook + Vault → K8s Secret 적용 | 높음 |
| **K8s 배포** | `kubectl apply -k` / ArgoCD | Playbook으로 apply, ArgoCD App 생성, 배포 후 검증 | 중간 |
| **Grafana 대시보드** | `apply-dashboard.sh` | `k8s` 모듈로 ConfigMap 적용 | 낮음 |
| **로컬/Dev Docker** | `start-all.sh`, 각 `compose-build.sh` | Playbook: 의존성·이미지·env·기동·migrate | 중간 |
| **DB 마이그레이션** | `docker exec ... prisma migrate deploy` / 수동 | Playbook 태스크 (Web Pod 또는 run job) | 중간 |
| **PostgreSQL 백업/복원** | `backup.sh`, `restore.sh`, K8s CronJob | Playbook으로 수동 백업/복원 절차 통일 | 낮음 |
| **배포 후 검증** | 수동 curl/kubectl | Playbook: health 엔드포인트·rollout 상태 확인 | 높음 |
| **노드/베어메탈 준비** | 수동 또는 별도 문서 | Playbook: Docker, kubeadm, NFS 클라이언트, 방화벽 등 | 환경별 |

---

## 2. Secrets 및 설정 적용

### 2.1 현재 방식

- **스크립트**: `infra/k8s/scripts/apply-secrets.sh`
  - `infra/docker/web/.env`, `infra/docker/ai/.env`에서 변수 로드
  - `k8s/base/secrets.yaml`의 `REPLACE_WITH_*` 치환 후 `kubectl apply`
- **수동**:
  - Harbor pull secret: `kubectl create secret docker-registry ...`
  - TLS secret: `kubectl create secret tls deepquest-tls-secret ...`

### 2.2 Ansible로 할 수 있는 것

- **Ansible Vault**에 비밀값 저장 후 Playbook에서:
  - `k8s` 모듈 또는 `community.kubernetes.kubernetes`로 Secret 리소스 생성/갱신
  - 또는 기존처럼 `secrets.yaml` 템플릿 치환 후 `kubectl apply` 실행
- Harbor pull secret, TLS 인증서도 변수로 넣어 한 번에 적용 가능
- **역할 예시**: `roles/deep-quest-secrets`
  - vars: Vault 암호화 변수 (Clerk, Supabase, DB, Google API, Harbor 등)
  - tasks: postgres-secret, ai-secret, web-secret, tailscale-auth, harbor-pull-secret, deepquest-tls-secret 생성/갱신

## 3. Kubernetes 배포 및 ArgoCD

### 3.1 현재 방식

- **직접 배포**: `kubectl apply -k k8s/overlays/dev` 또는 `prod`
- **ArgoCD**: `kubectl apply -f k8s/argocd/application-*.yaml` (수동 또는 한 번)
- **모니터링**: `kubectl apply -k k8s/monitoring/ingress-nginx-metrics/`, `apply-dashboard.sh`

### 3.2 Ansible로 할 수 있는 것

- **Playbook 1: K8s 매니페스트 적용**
  - `kustomize build` 또는 `kubectl apply -k`를 `command`/`shell` 또는 `k8s` 모듈로 실행
  - 환경(dev/prod)을 변수로 받아 overlay 선택
- **Playbook 2: ArgoCD Application**
  - ArgoCD Application YAML을 적용하는 태스크 (필요 시 한 번만 실행)
- **Playbook 3: 배포 후 검증**
  - `kubectl rollout status deployment/web-server -n deep-quest` (및 ai-server)
  - 클러스터 내에서 또는 Ingress URL로 `GET /api/health`, `GET /ok` 호출해 200 확인
- **Grafana 대시보드**: `apply-dashboard.sh` 내용을 Ansible `k8s` 또는 `command` 태스크로 대체 가능 (우선순위 낮음)

## 4. 로컬/Dev Docker 스택

### 4.1 현재 방식

- `infra/docker/start-all.sh`: 이미지 존재 여부 확인 → 없으면 `compose-build.sh` → `.env` 없으면 템플릿 복사 → `docker-compose up -d`
- `infra/docker/stop-all.sh`: `docker-compose down`
- 각 서비스별 `compose-build.sh`, `start.sh`, `stop.sh`

### 4.2 Ansible로 할 수 있는 것

- **Playbook: “Dev 호스트 한 대에 Deep Quest Docker 스택 준비”**
  - 대상: Docker·Docker Compose가 설치된 호스트 (또는 Ansible로 Docker 설치까지 포함)
  - 태스크 예:
    - `template`: `docker/postgres/.env`, `docker/ai/.env`, `docker/web/.env` (비밀은 Vault 또는 변수)
    - `copy` 또는 `template`: 필요한 설정 파일
    - `command`: `docker-compose -f docker-compose.yml build` (필요 시)
    - `command`: `docker-compose -f docker-compose.yml up -d`
    - 대기 후 `docker-compose run --rm web pnpm db:migrate:deploy` 실행
  - **역할 예시**: `roles/deep-quest-docker-dev`

## 5. DB 마이그레이션

### 5.1 현재 방식

- **Docker**: `docker-compose run --rm web pnpm db:migrate:deploy` 또는 `docker exec deepquest-web ...`
- **K8s**: `kubectl exec -n deep-quest deploy/web-server -- ...` 형태로 수동 실행

### 5.2 Ansible로 할 수 있는 것

- **Playbook 태스크**:
  - K8s: `community.kubernetes.kubernetes_exec` 또는 `command`: `kubectl exec -n deep-quest deploy/web-server -c web -- sh -c "cd /app && npx prisma migrate deploy"`
  - 실행 전에 rollout 상태 확인, 실행 후 exit code 확인
- 배포 파이프라인(Jenkins 등)에서 “이미지 배포 후 마이그레이션” 단계를 Ansible 플레이북 한 번 호출로 통일 가능

---

## 6. PostgreSQL 백업/복원

### 6.1 현재 방식

- **Docker**: `infra/docker/postgres/backup.sh`, `restore.sh`
- **K8s**: CronJob으로 정기 백업, 수동 복원은 문서/절차에 의존

### 6.2 Ansible로 할 수 있는 것

- **수동 백업/복원 절차의 일관화**
  - Playbook: “지금 시점 백업 수행” (Docker: backup.sh 호출 또는 pg_dump 직접 실행; K8s: Job 실행 또는 Pod에서 pg_dump)
  - Playbook: “특정 백업 파일로 복원” (절차 단계화 + 확인)
- CronJob 자체는 K8s 매니페스트로 두고, “한 번 수동 백업” / “재해 복구 절차”만 Ansible로 정리하는 방식도 가능

---

## 7. 배포 후 검증 (Health Check)

### 7.1 현재 방식

- 수동: `kubectl get pods -n deep-quest`, `curl` 로 `/api/health`, `/ok` 확인

### 7.2 Ansible로 할 수 있는 것

- **Playbook: “배포 검증”**
  - `kubectl rollout status deployment/web-server -n deep-quest`
  - `kubectl rollout status deployment/ai-server -n deep-quest`
  - Ingress URL이 있다면 `uri` 모듈로 `https://<ingress-host>/api/health`, `https://<ingress-host>/.../ok` (또는 내부 Service URL) 호출해 200 확인
  - 실패 시 명확한 실패 메시지로 빠르게 롤백/재시도 결정 가능

---

## 8. 베어메탈/노드 준비 (선택)

### 8.1 현재 방식

- 문서/회고에 따르면: 베어메탈 서버 → K8s, Harbor, Jenkins, NFS, Grafana 등은 별도 구축·관리
- 리포지토리에는 “OS부터 노드 준비”하는 스크립트는 없음

### 8.2 Ansible로 할 수 있는 것

- **새 노드 추가 시**
  - Docker 설치, kubeadm join (또는 기존 클러스터 가이드에 맞춤)
  - NFS 클라이언트 마운트, 방화벽, SSH 등 공통 보안 설정
- **역할 예시**: `roles/node-prep`, `roles/k8s-worker` (프로젝트 범위를 “앱 배포”로 한정하면 낮은 우선순위)

## 9. 권장 구현 순서

1. **Secrets 적용 Playbook + Vault**  
   - `apply-secrets.sh` 대체, Harbor/TLS까지 포함해 한 번에 적용·재현 가능하게 함.
2. **배포 후 검증 Playbook**  
   - rollout + health 엔드포인트 확인으로 배포 품질을 자동화.
3. **K8s 배포 Playbook**  
   - `kubectl apply -k` (또는 kustomize build 후 apply)를 환경 변수로 실행하고, 위 검증과 연결.
4. **DB 마이그레이션 태스크**  
   - K8s/Docker 공통으로 “한 번 실행” 형태의 Playbook으로 정리.
5. **로컬 Docker Dev Playbook**  
   - 새 PC/멤버 온보딩 시 `start-all.sh` 대신 Playbook 한 번으로 환경 구성.
6. **Grafana 대시보드 / 백업·복원**  
   - 필요 시 기존 스크립트를 Ansible 태스크로 감싸서 통일.

## 10. Ansible 구조 예시

```text
infra/ansible/
├── inventory/
│   ├── dev.yml
│   └── prod.yml
├── group_vars/
│   ├── dev/
│   │   └── vars.yml
│   └── prod/
│       └── vars.yml
├── playbooks/
│   ├── secrets-apply.yml    # Secrets + Harbor + TLS
│   ├── k8s-deploy.yml       # apply -k overlays/<env>
│   ├── deploy-verify.yml    # rollout + health
│   ├── db-migrate.yml       # Prisma migrate (K8s or Docker)
│   └── docker-dev.yml       # 로컬 Docker 스택 기동
├── roles/
│   ├── deep-quest-secrets
│   ├── deep-quest-k8s-deploy
│   └── deep-quest-verify
└── ansible.cfg
```

- **Vault**: `group_vars/*/vault.yml` (암호화)에 비밀 저장, `ansible-playbook --ask-vault-pass` 또는 vault password 파일로 실행.

## 11. 참고

- **Jenkins**: 현재 Jenkins가 빌드·Harbor 푸시·인프라 레포 이미지 태그 푸시까지 담당. “K8s에 apply” 또는 “Secrets 적용 + apply” 단계를 Ansible Playbook으로 옮기면 Jenkins는 `ansible-playbook` 한 번 호출로 정리 가능.
- **ArgoCD**: GitOps로 매니페스트는 Git이 소스이므로, Ansible은 “최초 ArgoCD Application 등록”, “Secrets처럼 Git에 넣지 않는 리소스 적용”, “검증”에 두는 구성이 자연스럽습니다.
