## 1. LangGraph Production 배포 (langgraph dev 탈출)

### 핵심
- 개발용 서버(`langgraph dev`)는 단일 프로세스/스레드 제약이 있음.
- 프로덕션 모드(uvicorn 기반 또는 공식 프로덕션 커맨드)로 전환해야 멀티 워커/스레드 활용 가능.

### Check: AI 이미지 빌드·배포 (2026-02 적용 완료)

**AS-IS (과거):**
- 커스텀 Dockerfile, `langgraph dev --port 8123` (inmem/SQLite).

**현재 (적용 완료):**
- **Jenkins:** `cd ai && uv run langgraph build -t harbor.../ai:${BUILD_TAG}` 후 Harbor 푸시.
- **K8s:** 공식 프로덕션 이미지 사용, **포트 8000** (containerPort, Service, ConfigMap LANGGRAPH_API_URL, Web wait-for-ai).
- **환경 변수:** REDIS_URI=redis://redis-service:6379, DATABASE_URI from ai-secret (LANGGRAPH_DATABASE_URI).

**체크포인트:** ✅ **완료** — `langgraph build` 이미지로 배포 중, Redis·Postgres 연동.

## 2. Redis 배포 및 연결 (State Store & Broker)

### 핵심
- in-mem 저장소를 Redis로 교체.
- 파드 재시작 시에도 작업 상태 유지, Redis가 비동기 큐 역할.

### Check: langgraph.json / 환경 변수

**현재 `ai/langgraph.json`:**
```json
{
  "dependencies": ["."],
  "graphs": { ... },
  "env": ".env",
  "image_distro": "wolfi"
}
```
- **POSTGRES_URI / REDIS_URI** 설정 없음.
- 체크포인트: `langgraph-checkpoint-sqlite` 사용 (로컬 SQLite).
- 런타임: `langgraph-runtime-inmem` (pyproject.toml 기준).

**현재 인프라 (2026-02 적용 완료):**
- Web/DB: PostgreSQL 사용 (Prisma, `postgres-service`).
- AI: **Redis 배포** (`redis-service:6379`), **REDIS_URI**·**DATABASE_URI**(ai-secret `LANGGRAPH_DATABASE_URI`) 설정, `langgraph build` 이미지로 프로덕션 배포.

**체크포인트:** ✅ **완료** — Redis 배포·연결 및 LangGraph 전용 DB(langgraph) 사용.

## 3. N_JOBS = 40 검증 (Concurrency)

### 핵심
- 설정만 40이어도 **AI 노드가 동기 함수(`def`)면** GIL 때문에 1~3개만 실질 동작할 수 있음 → **이전 “3개만 돌아감” 원인**.
- **필수:** 그래프 노드는 **`async def`** 로 작성되어 있어야 N_JOBS 병렬성이 살아남음.

### 검증: AI 노드 함수 정의

**검색 결과 (ai/src/graphs/**/*.py):**
- `resume_parser/nodes.py`: `def resume_parsing(...)`, `def _download_pdf_with_retry(...)` → **동기**
- `question_gen/nodes.py`: `def question_gen_node(...)` → **동기**
- `question_feedback_gen/nodes.py`: `def question_feedback_gen_node(...)`, `def structured_guide_answer_gen_node(...)` → **동기**
- `jd_to_text/nodes.py`: `def parse_jd_node(...)`, `def parse_jd_image_node(...)` → **동기**
- `jd_structuring/nodes.py`: `def jd_structuring(...)` → **동기**

**모든 그래프 노드가 `def`(동기)입니다. `async def` 노드는 없음.**

**체크포인트:** 부하 시 로그에 asyncio 스레드가 활발히 찍히는지, API 호출이 병렬로 나가는지 확인.

**현재:** ❌ **동기 함수만 사용** — N_JOBS=40이어도 **GIL로 인해 1~3개만 실질 동시 실행**될 수 있음. **노드를 `async def`로 바꾸고, 내부에서 `await` 가능한 비동기 클라이언트 사용**이 필요.

## 4. 메모리 증설 (Capacity Planning)

### 핵심
- 동시 실행 3개 → 40개로 늘리면 메모리 사용량 **최소 10배 이상** 증가 가능.
- OOMKilled 방지: 리밋을 넉넉히(예: 2Gi) 두고, 40개 동작 시 모니터링 후 Right-sizing.

### 현재 리소스 설정

**base (infra/k8s/base/ai/deployment.yaml):**
- requests: cpu `1`, memory `2Gi`
- limits: cpu `2`, memory `4Gi`

**prod overlay (infra/k8s/overlays/prod/kustomization.yaml):**
- requests: cpu `1`, memory `2Gi` 유지
- **limits 제거** (Task 2-1)

**patch-ai-deployment-resources.yaml (선택 적용):**
- requests: cpu `500m`, memory `2Gi`

**체크포인트:** 파드 메모리 limit 확인 (예: 512Mi 여부). OOMKilled 대비.

**현재:** ⚠️ **requests 2Gi, prod에서는 limits 없음** — 40개 동시 시 메모리 사용량 모니터링 필요. 초기에는 2Gi 이상 넉넉히 두고, 실제 사용량 보고 최적화 권장.

## 5. Queue 기반 HPA 설정 (KEDA 도입)

### 핵심
- CPU HPA만으로는 “CPU가 낮은데 대기열만 길어지는” 상황에 대응 불가.
- **KEDA**로 **Redis List 길이**(또는 LangGraph 내부 큐 메트릭)를 트리거로 사용.
- 예: “대기 중인 작업이 100개 넘으면 파드 1개 추가.”

### Check: KEDA / HPA 설정

**현재 HPA (infra/k8s/overlays/prod/patch-ai-hpa-tuning.yaml):**
- **Resource 메트릭만 사용:** CPU 35%, Memory 80%.
- **큐 길이 / Redis 메트릭 없음.**

**저장소 검색:** `KEDA`, `ScaledObject`, `ScaledJob`, Redis 트리거 — **미사용.**

**체크포인트:** KEDA ScaledObject 등으로 **Redis 큐 길이(또는 큐 메트릭)** 트리거가 적용되었는지.

**현재:** ❌ **KEDA 미도입** — CPU/메모리 HPA만 있음. Task 2 결론대로 하려면 KEDA + Redis(또는 큐) 메트릭 도입 필요.

## 권장 적용 순서 (가이드와 동일)

1. **LangGraph Production 배포**  
   - Dockerfile CMD를 프로덕션용 커맨드로 변경 (`langgraph up` 또는 공식 standalone/uvicorn 방식 확인 후 적용).
2. **Redis 배포 및 연결**  
   - Redis 배포 후, LangGraph에 REDIS_URI(및 필요 시 DATABASE_URI) 설정.
3. **N_JOBS=40 검증**  
   - 그래프 노드를 **`async def`** 로 전환하고, 부하 테스트로 동시 처리 수·asyncio 로그 확인.
4. **메모리 증설**  
   - 40개 동시 시 메모리 모니터링, 필요 시 requests/limits 상향(예: 2Gi 이상).
5. **KEDA 도입**  
   - CPU HPA 보조 또는 대체로, Redis 큐 길이(또는 큐 메트릭) 기반 스케일링 추가.
