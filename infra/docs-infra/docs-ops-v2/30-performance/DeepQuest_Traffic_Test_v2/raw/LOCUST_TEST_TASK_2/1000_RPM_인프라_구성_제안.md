## 1. 요구사항

**목표:**
- 2분 안에 1000개 처리
- 처리 시간: 2분 per request
- 목표 RPM: 1000 requests per minute

**동시 처리 필요량:**
- 2분 동안 1000개를 처리하려면 동시에 1000개가 실행되어야 함
- (처리 시간이 2분이므로)

---

## 2. 현재 상태 분석

**현재 설정:**
- `N_JOBS_PER_WORKER`: 40 (파드당 동시 처리 가능한 run 수)
- AI 파드 수: HPA로 스케일링 (min 1, max 10)
- 현재 최대 동시 처리: 10 파드 × 40 = 400개

**부족한 처리 능력:**
- 목표: 1000개 동시 처리
- 현재: 400개 동시 처리
- 부족: 600개 (150% 증가 필요)

---

## 3. 클러스터 리소스 분석

**현재 클러스터 리소스:**
- Master: 2 CPU, ~6GB Memory
- Worker 5개: 각각 2 CPU, ~8GB Memory
- 총 Worker 리소스: 10 CPU, ~40GB Memory

**파드당 리소스:**
- CPU: 1 core (request), 2 core (limit)
- Memory: 2Gi (request), 4Gi (limit)

**클러스터 제약:**
- CPU: 최대 10개 파드 (request 기준)
- Memory: 최대 20개 파드 (request 기준)
- 실제 제약: CPU (10개 파드가 한계)

---

## 4. 구성 옵션

### 옵션 A: 클러스터 확장 없이 최대한 활용 (즉시 적용 가능)

**설정:**
- `N_JOBS_PER_WORKER`: 40 → 100
- AI 파드 수: 10개 (HPA maxReplicas: 10)
- 동시 처리 능력: 10 × 100 = 1000개

**리소스 요구사항:**
- CPU: 10 core (request), 20 core (limit)
- Memory: 20Gi (request), 40Gi (limit)

**장점:**
- 클러스터 확장 없이 즉시 적용 가능
- 목표 달성 가능 (1000개 동시 처리)

**단점:**
- 파드당 부하 증가 (100개 동시 처리)
- 메모리 부족 가능성
- 파드 장애 시 영향 큼
- 클러스터 CPU 한계 도달 (확장 여지 없음)

**권장도:** 중간 (단기 해결책)

---

### 옵션 B: 클러스터 확장 (장기 권장)

**설정:**
- `N_JOBS_PER_WORKER`: 40 → 50
- AI 파드 수: 25개 (HPA maxReplicas: 25)
- 동시 처리 능력: 25 × 50 = 1250개 (25% 여유)

**리소스 요구사항:**
- CPU: 25 core (request), 50 core (limit)
- Memory: 50Gi (request), 100Gi (limit)

**필요한 클러스터 확장:**
- Worker 노드 추가: 최소 15개 CPU 추가 필요
- 예: Worker 8개 추가 (각 2 CPU) = 16 CPU 추가
- 총: 26 CPU (Master 제외)

**장점:**
- 균형잡힌 구성 (파드당 부하 적당)
- 안정성과 리소스 효율성 균형
- 25% 여유로 안정성 확보
- HPA로 자동 스케일링 가능
- 확장 여지 있음

**단점:**
- 클러스터 확장 필요 (비용 증가)
- 인프라 변경 필요

**권장도:** 높음 (장기 해결책)

---

### 옵션 C: 극단적 최적화 (비권장)

**설정:**
- `N_JOBS_PER_WORKER`: 40 → 200
- AI 파드 수: 5개
- 동시 처리 능력: 5 × 200 = 1000개

**리소스 요구사항:**
- CPU: 5 core (request), 10 core (limit)
- Memory: 10Gi (request), 20Gi (limit)

**장점:**
- 리소스 절약

**단점:**
- 파드당 부하 과다 (200개 동시 처리)
- 불안정성 높음
- 메모리 부족 위험 높음
- 파드 장애 시 영향 매우 큼

**권장도:** 낮음 (위험함)

---

## 5. 단계별 적용 방안

### 1단계: 즉시 적용 (클러스터 확장 전)

**목표:** 클러스터 확장 없이 목표 달성

**설정 변경:**
- `N_JOBS_PER_WORKER`: 40 → 100
- HPA maxReplicas: 10 유지
- HPA minReplicas: 1 → 3 (기본 부하 대비)
- HPA CPU 기준: 35% 유지

**예상 결과:**
- 동시 처리: 10 × 100 = 1000개

- 제약: 클러스터 CPU 한계 도달

---

### 2단계: 클러스터 확장 후 (장기)

**목표:** 안정성과 확장성 확보

**클러스터 확장:**

- 동시 처리: 25 × 50 = 1250개 (25% 여유)
- 리소스: CPU 25 core, Memory 50Gi
- 확장 여지: 여전히 여유 있음

**적용 파일:**

## 8. inmem 런타임 유지 가능 여부

### 8.1 현재 런타임

**사용 중인 런타임:**
- `langgraph-runtime-inmem`: in-memory 큐 기반
- `langgraph-checkpoint-sqlite`: 체크포인트 저장 (그래프 상태)
- Redis: 사용 안 함

**차이점:**
- **체크포인트 (Checkpoint)**: 그래프 실행 상태 저장 (SQLite 사용)
- **런타임 큐**: 실행 요청 큐 (inmem 사용, 메모리 기반)

---

### 8.2 inmem 유지 시 장점

**1. 간단한 설정:**
- 추가 인프라 불필요 (Redis 클러스터 등)
- 빠른 시작 및 운영 복잡도 낮음
- 비용 절감

**2. 1000개 목표 달성 가능:**
- `N_JOBS_PER_WORKER`: 40 → 100
- 파드 수: 10개
- 동시 처리: 10 × 100 = 1000개
- 목표 달성 가능

**3. 현재 단계에 적합:**
- MVP 단계에서는 충분함
- 추가 인프라 구축 비용 없음

---

### 8.3 inmem 유지 시 단점

**1. 파드별 독립적인 큐:**
- 파드 A의 큐와 파드 B의 큐가 분리됨
- 부하 분산이 완벽하지 않을 수 있음
- Session affinity로 인해 특정 파드에 부하 집중 가능

**2. 파드 재시작 시 큐 손실:**
- 파드가 재시작되면 큐에 있던 run이 손실됨
- 처리 중이던 run도 손실 가능
- 체크포인트는 SQLite에 저장되지만, 큐에만 있고 아직 처리 안 된 run은 손실

**3. 확장성 제한:**
- 파드 수에 의존
- 클러스터 리소스 제약
- 더 많은 동시 실행을 원하면 파드 수 증가 필요

**4. 메모리 제한:**
- 큐가 파드 메모리에 저장됨
- 큐가 너무 길어지면 메모리 부족 가능
- 파드당 메모리 제한 (4Gi) 내에서 동작

**5. 모니터링 제한:**
- 큐 길이를 직접 모니터링하기 어려움
- 파드별 큐 상태 확인 필요

---

### 8.4 Redis 기반 런타임 전환 시 장점

**1. 분산 큐:**
- 모든 파드가 같은 큐를 공유
- 부하 분산이 더 효율적
- 파드 재시작 시에도 큐 유지

**2. 확장성:**
- 파드 수와 무관하게 큐 관리
- 더 많은 동시 실행 가능

**3. 내구성:**
- Redis 지속성 설정 가능
- 큐 손실 방지

**4. 모니터링:**
- Redis 큐 길이 모니터링 가능
- 중앙 집중식 관리

**단점:**
- 추가 인프라 필요 (Redis 클러스터)
- 설정 복잡도 증가
- 비용 증가

---

### 8.5 결론 및 권장 사항

**당장은 inmem 유지 가능:**

- 1000개 목표 달성 가능
- 추가 인프라 불필요
- 설정 간단
- 비용 절감

**주의사항:**
- 파드 재시작 시 큐 손실 가능 (체크포인트는 유지되지만 큐에만 있는 run은 손실)
- 부하 분산이 완벽하지 않을 수 있음
- 더 많은 동시 실행을 원하면 파드 수 증가 필요

**Redis 전환 고려 시점:**
1. 1000개 이상의 동시 실행 필요 시
2. 파드 재시작 시 큐 손실이 문제가 될 때
3. 더 나은 부하 분산이 필요할 때
4. 프로덕션 안정성이 중요할 때
5. 큐 모니터링이 필요할 때

**현재 단계 권장:**
- inmem 유지 (1000개 목표 달성 가능)
- 1단계 설정 적용 (`N_JOBS_PER_WORKER`: 40 → 100)
- 테스트 후 문제 발생 시 Redis 전환 검토
