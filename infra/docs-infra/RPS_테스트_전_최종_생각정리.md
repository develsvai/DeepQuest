# DeepQuest RPS 테스트 전 최종 생각정리

## 배포작업 하면서 느낀점

- 프로젝트 구조 파악이 중요
- SaaS는 괜히 SaaS가 아니다
  - 온프렘에서 직접 배포 하려면 고려해야 될 게 한두 개가 아님

## 충분히 고려 해야 될 점

### env
- 특히 중요. 대부분의 문제들을 파고 들어가 보면 근본적으로 env 문제임

### Clerk
- Webhook 키 잘 확인

### Sentry
- 여전히 설정이 먹었는지 아닌지는 모르겠지만
- 프로덕션 환경에서 에러의 범위를 좁히는 데 많은 기여를 함
  - 이후 다른 프로젝트를 한다면 충분히 고려해볼 대상

### prod, dev
- 환경별로 키값 또는 설정을 분리하는 게 중요
  - 특히 Clerk 같은 애들은 보안 설정이 많아서 dev에서 prod 같이 쓰면 여간 귀찮은 게 아닐 거 같음
- HPA, HA 구성 시 dev와 prod의 리소스 할당 값이 같으면 안 됨
  - dev는 어디까지고 dev 환경임

## 개선이 필요한 점

### GitOps 파이프라인
- 현재 구성에서 동작은 잘 하는데, 현재 인프라 구조에 적합한 구성인지 확인 필요

### tRPC
- 내가 잘 몰라서 그런 거 같기도 한데, tRPC 쪽 예외나 유저 정보 처리 부분에서 부족한 점들이 보였음 → 확인 필요

### 현재 Jenkins 환경
- 현재 deploy 브랜치를 따로 두고, infra repo에서 선택된 브랜치를 따라가게 해뒀는데
- 이게 맞는 선택인지 잘 모르겠음

### 앤서블 자동화
- 일단 문서를 작성해 놓았음
- 테라폼도 마찬가지로 포폴 나오고 생각

### 문서 작성
- AI가 문서를 많이 만들어냄
- 문서를 작성하는 룰이 있어야 할 듯
- 문서를 그냥 막 생성하니까:
  - 언제 만들어진 건지 모르겠고
  - 뭔 내용이 있는지 모르겠고
  - 최신 문서인지도 모르겠음
  - 정리할 때가 너무 힘듦
    - 모든 문서를 다 읽어야 하고
    - 중복 제거해야 하고
    - 또 다시 분류해야 함
- **현재 문서 작성룰 기반 경로 구분:**
  - `docs/canonical/` — 항상 신뢰 가능한 기준 문서
  - `docs/evolution/` — 변경 이력 / 사건 / 작업 흐름
  - `docs/troubleshooting/` — 장애 유형별 해결 지식
  - `docs/archive/` — 개인 메모 / 실험 기록 / 참고용

## (추가) RPS 테스트 전 꼭 점검해볼 것들

- **헬스체크 / 라이브니스**
  - K8s `livenessProbe`, `readinessProbe`가 실제 트래픽을 받을 수 있는 기준과 맞는지
  - RPS가 올라갈 때 헬스체크가 실패해서 Pod가 계속 재시작되면 안 됨
- **커넥션 풀**
  - DB 커넥션 풀 크기 vs Pod 수 vs 동시 요청 수
  - RPS가 높아지면 커넥션 고갈로 에러가 날 수 있음
- **로그 볼륨**
  - RPS가 높으면 로그가 폭증함. 로그 레벨·샘플링·로테이션 확인
- **리소스 한계**
  - CPU/메모리 `requests`, `limits`가 RPS 증가에 맞게 설정돼 있는지
  - OOMKilled, CPU Throttling 발생 여부 미리 점검
- **외부 의존성**
  - Clerk, Supabase, AI 서버 등 호출 시 타임아웃·재시도 정책
  - 한쪽이 느려지면 전체 레이턴시가 올라가는 구조인지 파악

## (추가) 문서 작성 룰 보강 아이디어

- **메타데이터 필수**
  - 상단에 `업데이트 날짜`, `작성자`, `분류(canonical/evolution/troubleshooting/archive)` 명시
- **생성 규칙**
  - 새 문서 작성 시 먼저 `분류`를 정하고, 해당 경로에만 저장
  - AI가 문서 생성할 때 “이 문서의 목적과 분류”를 먼저 정의하게 유도
- **아카이브 기준**
  - 3개월 이상 미수정 + 참고용 문서 → `archive/`로 이동
  - `canonical`은 변경 시 리뷰·승인 절차 권장
- **중복 방지**
  - 비슷한 주제 문서 생성 전, 기존 `canonical` / `troubleshooting` 검색 후 갱신 vs 신규 작성 판단
