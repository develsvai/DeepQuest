# 작업 기록 갱신

## 목표

가장 최신 `docs/tasks/task-YYYY-MM-DD.md`를 실제 인프라 실행 상태와 일치시키고, 특히 `진행`과 `보류 / 다음 액션`을 현재 기준으로 정리한다.
또한 오늘 `task`의 완료 결과가 다음날 `status`로 자연스럽게 승격될 수 있도록, 큰 작업 축과 완료 여부가 읽히는 형태로 남기는 것이 목표다.

## 작업 순서

1. `docs/tasks/README.md`와 가장 최신 `task-YYYY-MM-DD.md`를 읽는다
2. 필요하면 가장 최신 `docs/status/status-YYYY-MM-DD.md`도 함께 읽는다
3. 실제로 완료된 일, 검증된 사실, 중단 지점을 정리한다
4. 아래 섹션을 현재 상태로 갱신한다
   - `진행`
   - `보류 / 다음 액션`
5. 필요할 때만 `계획`을 최소 범위로 보정한다

## 규칙

- 이 skill의 주 대상은 `진행`이다
- `계획`은 이미 세운 인프라 작업 방향이 흔들리지 않도록 유지하고, 실제 작업 때문에 문맥 보정이 필요한 경우에만 최소 수정한다
- 아직 하지 않은 일을 `진행`에 쓰지 않는다
- 추측이 아니라 실제로 검증된 실행 내용만 적는다
- task 문서의 현재 구조와 헤더 형식을 유지한다
- 24일 인프라 task 문서처럼 큰 작업 단위 아래에 핵심 내용을 정리하는 방식을 우선 따른다
- 어떤 계획 항목이 끝났는지 `진행`에서 번호나 작업 축 기준으로 읽히게 적는다
- 완료된 계획 항목은 `보류 / 다음 액션`에서 제거하고, 남은 항목만 다음날로 넘길 수 있게 정리한다
- 새 이슈나 작업 축 전환이 생기면 `진행`에 먼저 근거를 남긴 뒤 `보류 / 다음 액션`을 바꾼다
- 다음날 `status`가 이 문서를 읽고 `현재 상태`, `알려진 이슈`, `다음 작업`을 만들 수 있도록, `진행`은 완료 사실 중심으로 쓰고 `보류 / 다음 액션`은 미완료 항목만 남긴다

## 작성 스타일

- `진행`은 시간순 명령 실행 로그가 아니라 작업 맥락 요약으로 작성한다
- 같은 문제를 다룬 내용은 `### 1. 현재 구조 확인`처럼 작업 단위 제목 아래에 묶는다
- 각 작업 단위에는 아래 항목이 읽히게 한다
  - 어떤 문제가 있었는지
  - 어떤 핵심 증거로 판단했는지
  - 무엇을 고쳤거나 확인했는지
  - 남은 위험이나 다음 후보가 있는지
- 명령어, Pod 이름, 긴 로그 라인, 반복 확인 결과는 task에 모두 적지 않는다
- task에는 나중에 작업 맥락을 복원하는 데 필요한 식별자와 결론만 남긴다
- 상세 증거와 원본성 있는 값은 `docs/infra_state/`나 `docs/troubleshooting/` 문서에 남기고 task에서는 해당 문서를 참조한다
- task에 문서 링크를 남길 때는 작성 중인 task 파일 위치 기준 상대경로로 쓴다
- 다른 문서 디렉터리는 `../infra_state/file.md`, `../troubleshooting/file.md`처럼 작성하고, 로컬 절대경로 링크는 쓰지 않는다
- 한 작업 단위가 5~7개 bullet을 넘기면 불필요한 실행 세부사항이 섞인 것으로 보고 압축한다
- 전체 `진행`이 길어질 때는 새로 알게 된 사실을 무조건 추가하지 말고, 기존 bullet을 더 정확한 결론으로 갱신한다
- `보류 / 다음 액션`에는 미완료된 실제 행동만 남기고, 이미 끝난 확인이나 배경 설명은 넣지 않는다
