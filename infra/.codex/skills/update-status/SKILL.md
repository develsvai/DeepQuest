# 상태 갱신

## 목표

이 workflow에서는 `status`를 하루 시작 기준 문서로 유지하고, 당일 진행 추적은 `task`로 넘긴다. 따라서 `update-status`는 기본 흐름에서 사용하지 않는 예외용 skill이다.

사용 조건:

- 장애나 외부 사건으로 당일 작업 의도 자체가 바뀌었다
- 기존 `status`의 `다음 작업`이 더 이상 최신 우선순위를 설명하지 못한다
- `task`만 갱신하면 다음 작업일의 `new-status`가 잘못된 방향을 이어받을 가능성이 있다

## 작업 순서

1. 가능하면 먼저 `update-task`로 당일 진행 내용을 정리한다
2. 정말 예외적으로 `status`를 수정해야 하면, 최신 `status`와 최신 `task`를 함께 읽는다
3. 기존 시작 시점 기록은 지우지 않고, 필요한 최소 문장만 보충한다

## 규칙

- 기본값은 `status`를 다시 쓰지 않는 것이다
- 단순 진행, 완료, 중단, 세부 판단은 `status`가 아니라 `task`에 쓴다
- 당일 진행, 판단, 중단 지점, 실행 결과는 `task`에 남긴다
- `status`는 새 작업일 시작 시 `new-status`가 전날 `status`와 최신 `task`를 바탕으로 다시 만든다
- 예외적으로 `status`를 손볼 때도, 당일 `task`의 `진행`과 `보류 / 다음 액션`으로 설명되지 않는 메타 설명을 덧붙이지 않는다
- `status`의 `현재 상태`는 확인된 사실만 최소 보정하고, `다음 작업`은 같은 날 `task`의 큰 작업 축과 어긋나지 않게 유지한다
- status에 문서 링크를 남길 때는 작성 중인 status 파일 위치 기준 상대경로로 쓴다
- 다른 문서 디렉터리는 `../tasks/file.md`, `../infra_state/file.md`처럼 작성하고, 로컬 절대경로 링크는 쓰지 않는다
