## 1. Ingress 메트릭이 Prometheus에 안 뜰 때

10254 포트에서는 메트릭이 나오는데 **Prometheus Explore에서는 안 나오면** Prometheus가 해당 타겟을 스크랩하지 않는 상태다. 아래 2~5번 순서로 원인을 좁힌다.

---

## 2. 해결 과정 요약 (현재 적용된 방식)

다음 두 가지를 적용하면 파일만으로 Ingress 메트릭이 Prometheus·Grafana에 나온다.

**1) Ingress Controller에서 메트릭 노출 켜기**

- Controller 기본값은 메트릭이 꺼져 있어 10254에 `nginx_ingress_controller_*` 가 없을 수 있다.
- Helm(ingress-nginx 차트) 사용 시 프로젝트의 values 오버라이드로 활성화. [6. Ingress Controller에서 메트릭 활성화](#6-ingress-controller에서-메트릭-활성화) 절차대로 적용.

**2) ServiceMonitor의 release 라벨을 Prometheus에 맞춰 두기 (파일에 반영)**

- Prometheus는 `serviceMonitorSelector.matchLabels.release`와 같은 라벨을 가진 ServiceMonitor만 스크랩한다. 값이 다르면 타겟이 생성되지 않는다.
- **현재 적용:** `ingress-nginx-metrics/kustomization.yaml` 의 `commonLabels.release` 에 클러스터 Prometheus가 선택하는 값(기본 `kube-prometheus-stack`)을 두고, `kubectl apply -k ingress-nginx-metrics/` 만 실행하면 된다. 스크립트로 패치할 필요 없음.
- Prometheus가 다른 release(예: `prometheus`)를 쓰면 `kustomization.yaml` 에서 `commonLabels.release` 만 해당 값으로 수정한 뒤 다시 적용하면 된다.

적용 순서: 1) Helm upgrade 로 메트릭 활성화 → 2) `kubectl apply -k ingress-nginx-metrics/` 로 Service·ServiceMonitor 적용. 1~2분 후 Prometheus Status → Targets 에 ingress-nginx 타겟이 생기고 Explore에서 `nginx_ingress_controller_*` 가 조회되어야 한다.

## 3. 타겟 등록 여부 확인

- Prometheus UI → **Status → Targets** 로 이동.
- 목록에서 `ingress-nginx` 또는 `10254` 가 포함된 타겟을 찾는다.
- **없음:** ServiceMonitor가 Prometheus에 선택되지 않음 → 4번으로 이동.
- **있는데 down / 에러:** 해당 타겟 클릭해 에러 메시지 확인 (네트워크, TLS, 타임아웃 등).

## 4. ServiceMonitor 라벨 맞추기

Prometheus는 `serviceMonitorSelector`에 맞는 라벨이 있는 ServiceMonitor만 스크랩한다. 라벨이 다르면 타겟이 아예 생성되지 않는다.

**방법 A: 파일로 적용 (권장)**

- `infra/k8s/monitoring/ingress-nginx-metrics/kustomization.yaml` 의 `commonLabels.release` 를 클러스터 Prometheus가 선택하는 값과 동일하게 둔다. kube-prometheus-stack 이면 기본값 `kube-prometheus-stack` 그대로 두고, 다르면 수정 후 적용한다.

```bash
# Prometheus가 요구하는 release 값 확인
kubectl get prometheus -n monitoring -o jsonpath='{.items[0].spec.serviceMonitorSelector.matchLabels.release}'
echo ""
# 위에서 나온 값으로 kustomization.yaml 의 commonLabels.release 수정 후
cd infra/k8s/monitoring
kubectl apply -k ingress-nginx-metrics/
```

**방법 B: 수동 패치 (한 번만 맞출 때)**

```bash
# Prometheus가 요구하는 release 값 확인
kubectl get prometheus -n monitoring -o jsonpath='{.items[0].spec.serviceMonitorSelector.matchLabels.release}'
echo ""

# 위에서 나온 값(예: prometheus)으로 ServiceMonitor 패치
kubectl patch servicemonitor -n monitoring ingress-nginx-controller-metrics --type=merge -p '{"metadata":{"labels":{"release":"prometheus"}}}'
```

**방법 C: 스크립트로 확인·패치**

```bash
cd infra/k8s/monitoring/ingress-nginx-metrics
chmod +x fix-servicemonitor-for-prometheus.sh
./fix-servicemonitor-for-prometheus.sh
```

패치 후 1~2분 지나면 Status → Targets 에 ingress-nginx 타겟이 생기고, Explore에서 `nginx_ingress_controller_*` 가 조회되어야 한다. Grafana 데이터소스가 이 Prometheus를 바라보고 있으면 대시보드에도 반영된다.

**그래도 안 나올 때**

- Grafana에서 사용하는 **데이터소스**가 위에서 확인한 Prometheus 인스턴스가 맞는지 확인.
- Prometheus가 여러 개면 (예: prometheus-oper-prometheus vs kube-prometheus-stack-prometheus) 올바른 쪽의 Targets를 확인한다.

## 5. Prometheus에 메트릭이 없을 때 상세 점검

아래 순서대로 확인하면 원인을 좁힐 수 있다.

**1. 메트릭 리소스 적용**

```bash
cd infra/k8s/monitoring
kubectl apply -k ingress-nginx-metrics/
```

**2. Service와 Endpoints 존재·라벨 확인**

```bash
kubectl get svc -n ingress-nginx ingress-nginx-controller-metrics -o wide
kubectl get endpoints -n ingress-nginx ingress-nginx-controller-metrics -o yaml
```

- `Endpoints`에 주소가 없으면 Service의 selector가 Controller Pod 라벨과 맞지 않는 것이다.
- Controller Pod 라벨 확인: `kubectl get pods -n ingress-nginx -L app.kubernetes.io/name,app.kubernetes.io/component,app.kubernetes.io/instance`
- Service는 `app.kubernetes.io/name=ingress-nginx`, `component=controller`, `instance=ingress-nginx` 로 맞춰져 있으므로, 설치한 Ingress Helm 차트의 Pod 라벨과 일치하는지 확인한다.

**3. 컨트롤러가 10254에서 메트릭 노출 여부**

```bash
# 클러스터 내부에서 metrics 엔드포인트 호출
kubectl run -it --rm debug --image=curlimages/curl --restart=Never -- curl -s "http://ingress-nginx-controller-metrics.ingress-nginx.svc.cluster.local:10254/metrics" | grep -E "nginx_ingress_controller_requests_total|nginx_ingress_controller_request_duration"
```

- 한 줄이라도 나오면 메트릭은 노출된 것이고, **Prometheus가 이 타겟을 스크랩하지 않는** 문제이므로 4번으로 간다.

**4. Prometheus가 이 ServiceMonitor를 스크랩하는지 확인**

- Prometheus UI → **Status → Targets** 에서 `ingress-nginx` 또는 `metrics` 가 포함된 타겟이 있는지 본다.
- **없다면** Prometheus가 `ingress-nginx-controller-metrics` ServiceMonitor를 선택하지 않는 것이다.  
  **원인:** ServiceMonitor의 `release` 라벨이 사용 중인 Prometheus의 `serviceMonitorSelector`(예: `release: kube-prometheus-stack`)와 다를 수 있다.
- **해결:** [4. ServiceMonitor 라벨 맞추기](#4-servicemonitor-라벨-맞추기) 대로 kustomization 의 `commonLabels.release` 를 맞추거나 패치/스크립트로 수정한다. 수정 후 1~2분 지나면 Targets에 ingress 관련 타겟이 생기고 Explore에서 `nginx_ingress_controller_*` 가 나와야 한다.

**5. 10254에서 `nginx_ingress_controller_*` 메트릭이 안 나올 때**

- Prometheus가 스크랩까지 하는데도 `nginx_ingress_controller_requests_total` 등이 비어 있으면 **컨트롤러가 해당 메트릭을 노출하지 않는** 경우다.
- 클러스터에서 10254 메트릭 직접 확인:

```bash
# 로컬에서 10254 포트 포워드
kubectl port-forward -n ingress-nginx svc/ingress-nginx-controller-metrics 10254:10254
# 다른 터미널에서 메트릭 이름 목록 확인
curl -s http://127.0.0.1:10254/metrics | grep "^# HELP" | sed 's/# HELP \([^ ]*\).*/\1/' | sort -u
```

- 목록에 `nginx_ingress_controller_requests_total`, `nginx_ingress_controller_request_duration_seconds_bucket` 가 **없으면** Ingress Controller 설정에서 HTTP 요청 메트릭을 켜야 한다. [6. Ingress Controller에서 메트릭 활성화](#6-ingress-controller에서-메트릭-활성화) 절차대로 적용한다.
- 공식 [ingress-nginx](https://github.com/kubernetes/ingress-nginx) 컨트롤러는 `--enable-metrics=true` 일 때 위 메트릭을 노출한다. 다른 Ingress 구현체를 쓰는 경우 해당 문서에서 Prometheus 메트릭 활성화 방법을 확인한다.

## 6. Ingress Controller에서 메트릭 활성화

클러스터에 Ingress Nginx Controller가 **Helm(ingress-nginx 차트)** 으로 설치되어 있으면, 아래 values 오버라이드로 메트릭을 켤 수 있다.

**1. 프로젝트에 포함된 values 파일**

- **경로:** `infra/k8s/ingress-nginx-metrics-values.yaml` (프로젝트 루트 기준)
- **내용:** `controller.metrics.enabled: true`, Pod 어노테이션 `prometheus.io/scrape`, `prometheus.io/port`

**2. 적용 (Helm upgrade)**

Release 이름·네임스페이스는 실제 설치와 맞게 바꾼다. 예: release `ingress-nginx`, namespace `ingress-nginx`.

```bash
# 저장소 추가 (한 번만)
helm repo add ingress-nginx https://kubernetes.github.io/ingress-nginx
helm repo update

# 메트릭 활성화 적용 (프로젝트 루트에서)
helm upgrade ingress-nginx ingress-nginx/ingress-nginx \
  --namespace ingress-nginx \
  -f infra/k8s/ingress-nginx-metrics-values.yaml
```

- 이미 사용 중인 values 가 있으면 기존 파일에 이 설정을 합치거나, `-f 기존.yaml -f infra/k8s/ingress-nginx-metrics-values.yaml` 처럼 같이 넘긴다.

**3. 적용 후 확인**

1~2분 후 컨트롤러 Pod가 재시작되면 10254 메트릭에 `nginx_ingress_controller_*` 가 나와야 한다.

```bash
kubectl port-forward -n ingress-nginx svc/ingress-nginx-controller-metrics 10254:10254
# 다른 터미널
curl -s http://127.0.0.1:10254/metrics | grep nginx_ingress_controller_
```

나온 뒤 [4. ServiceMonitor 라벨 맞추기](#4-servicemonitor-라벨-맞추기) 대로 `release` 를 Prometheus에 맞추면 Prometheus·Grafana에서 Ingress RPS/P95 패널에 데이터가 채워진다.

## 7. Grafana Ingress RPS & P95 패널 No data

**Prometheus Explore에서 `nginx_ingress_controller_requests_total` (또는 `nginx_ingress_controller_nginx_process_requests_total`) 쿼리 결과가 비어 있으면** Ingress 메트릭이 아직 수집되지 않는 상태다. [1~5번](#1-ingress-메트릭이-prometheus에-안-뜰-때) 절차(메트릭 리소스 적용 → Endpoints → 10254 노출 확인 → ServiceMonitor 라벨 맞추기)를 먼저 진행한다.

메트릭은 수집되는데 Grafana만 No data인 경우:

1. **Prometheus에서 메트릭 존재 확인**  
   Grafana가 바라보는 Prometheus → Explore에서 아래 쿼리 실행.
   - `nginx_ingress_controller_requests_total`
   - `nginx_ingress_controller_nginx_process_requests_total` (컨트롤러 전체 RPS용으로 사용하는 메트릭)
   - `{__name__=~".*ingress.*requests_total"}` (이름이 다르면 여기서 확인)
   - 결과가 비어 있으면 Ingress 메트릭이 수집되지 않는 상태다.

2. **ServiceMonitor 적용 여부**

```bash
kubectl get servicemonitor -n monitoring -l app.kubernetes.io/name=ingress-nginx
kubectl get svc -n ingress-nginx ingress-nginx-controller-metrics
```

ServiceMonitor와 metrics Service가 있어야 Prometheus가 10254 포트를 스크랩한다.

3. **컨트롤러 메트릭 노출 확인**  
   클러스터 안에서 컨트롤러 Pod의 10254 포트에 요청해 본다.

```bash
kubectl run -it --rm debug --image=curlimages/curl --restart=Never -- curl -s http://ingress-nginx-controller-metrics.ingress-nginx.svc:10254/metrics | head -50
```

`nginx_ingress_controller_*` 문자열이 보이면 메트릭은 노출된 것이고, Prometheus 스크랩/라벨 불일치 쪽을 확인한다.

4. **Prometheus scrape 대상 포함 여부**  
   Prometheus UI → Status → Targets 에서 `ingress-nginx` 또는 `metrics` 관련 타겟이 있고 **up** 인지 확인한다. 없거나 down 이면 ServiceMonitor의 `release` 라벨이 Prometheus의 selector와 맞는지 [4번](#4-servicemonitor-라벨-맞추기) 대로 확인한다.

## 8. 클러스터 확인 결과 참고

다음 항목은 한 번 확인해 둔 상태다. 환경이 바뀌면 다시 확인하면 된다.

| 항목 | 결과 |
|------|------|
| Service `ingress-nginx-controller-metrics` | 존재 (ingress-nginx NS, 10254) |
| Endpoints | 10.244.x.x:10254 로 Controller Pod 연결됨 (활성화 후 Pod 재시작으로 IP 갱신됨) |
| ServiceMonitor `ingress-nginx-controller-metrics` | 존재 (monitoring NS). 스크랩되려면 `release` 라벨이 Prometheus의 serviceMonitorSelector와 일치해야 함. |
| Prometheus `serviceMonitorSelector` | `release: kube-prometheus-stack` 등 사용 시, kustomization 의 `commonLabels.release` 를 같은 값으로 두면 스크랩 대상에 포함됨. |
| 10254 `/metrics` 응답 | 정상. 메트릭 활성화 적용 후 `nginx_ingress_controller_*` 다수 노출됨. |
| `nginx_ingress_controller_*` 메트릭 | admission_*, build_info, config_*, nginx_process_* (requests_total, connections 등), ssl_*, success 등 확인됨. RPS/지연용 `nginx_ingress_controller_requests_total`, `request_duration_seconds_bucket` 는 트래픽이 들어온 뒤 또는 컨트롤러 버전에 따라 노출될 수 있음. `nginx_ingress_controller_nginx_process_requests_total` 는 컨트롤러 전체 요청 수로 대시보드 RPS 대체용으로 사용 가능. |

## 9. Deep Quest 대시보드 패널 No data / 데이터소스

Deep Quest 전용 대시보드(`dashboards/deep-quest.json`)에서 패널이 전부 No data이거나 일부만 나올 때 아래를 순서대로 확인한다.

**1. 데이터소스 선택**

- 대시보드 상단 변수 **Data source**에서 사용할 Prometheus 데이터소스를 선택했는지 확인한다. 데이터소스가 비어 있거나 잘못된 대상이면 모든 패널이 No data가 된다.

**2. namespace=deep-quest 메트릭**

- 대부분의 패널은 `namespace="deep-quest"` 로 필터한다. Prometheus Explore에서 예시 쿼리가 결과를 반환하는지 확인한다.
  - `kube_deployment_status_replicas_ready{namespace="deep-quest"}`
  - `container_cpu_usage_seconds_total{namespace="deep-quest"}`
- 결과가 비어 있으면 해당 메트릭을 스크랩하는 구성(kube-state-metrics, cAdvisor 등)과 deep-quest 네임스페이스 Pod/Deployment 존재 여부를 확인한다.

**3. kube-state-metrics (HPA / Deployment / Pod 상태)**

- Ready Pods, HPA Replicas, Total Pods Status, Pod Restarts 패널은 `kube_*` 메트릭을 사용한다. kube-state-metrics가 클러스터에 설치·스크랩되고 있어야 값이 나온다. Prometheus Targets에 kube-state-metrics 타겟이 있고 up 인지 확인한다.

**4. Ingress RPS / HTTP Status Codes**

- Ingress RPS (Traffic Volume), HTTP Status Codes 패널은 `nginx_ingress_controller_requests_total` 를 사용한다. 이 메트릭이 수집되지 않으면 [1~6번](#1-ingress-메트릭이-prometheus에-안-뜰-때)(Ingress 메트릭 활성화·ServiceMonitor 적용) 절차를 먼저 진행한다.

**5. 적용 스크립트·ConfigMap**

- `dashboards/apply-deep-quest-dashboard.sh` 로 적용한 경우 ConfigMap `deep-quest-dashboard`(monitoring NS)에 `grafana_dashboard=1` 라벨이 있어야 Grafana sidecar가 대시보드를 로드한다. 수동 import 시에는 Grafana → Dashboards → Import → `dashboards/deep-quest.json` 업로드 후 데이터소스만 선택하면 된다.
