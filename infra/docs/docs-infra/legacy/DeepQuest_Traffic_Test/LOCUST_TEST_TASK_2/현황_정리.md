## 2. 적용된 작업 요약

### 2.1 빌드·배포

- **Jenkinsfile:** AI 빌드를 `langgraph build` + Harbor push로 변경. 에이전트는 SSH 스테이션(`ssh station`), Python 3.12 + uv 사용.
- **이미지:** `harbor.192.168.0.110.nip.io/deep-quest/ai:<BUILD_TAG>` (예: `141-65141ab`), `prod` 태그 병행 푸시.
- **K8s:** base·overlay 모두 **포트 8000** (containerPort, Service, ConfigMap LANGGRAPH_API_URL, Web wait-for-ai). REDIS_URI, DATABASE_URI(ai-secret) 적용.

### 2.2 Redis·Postgres

- **Redis:** `infra/k8s/base/redis/` Deployment 1 replica, Service `redis-service:6379`. AI deployment에 initContainer `wait-for-redis` 추가.
- **LangGraph DB:** 동일 Postgres 인스턴스에 DB `langgraph` 생성. `LANGGRAPH_DATABASE_URI`는 **ai-secret**에만 두고, `apply-secrets.sh`로 반영 (ai/.env.prod 기준).

### 2.3 시크릿·라이선스

- **ai-secret:** `LANGSMITH_API_KEY`, `LANGCHAIN_API_KEY`, `LANGGRAPH_DATABASE_URI`, `GOOGLE_API_KEY` 등. `apply-secrets.sh`는 **WEB → AI** 순으로 .env를 source하여 AI 값이 덮어쓰이지 않도록 함.
- **License:** langgraph-api 0.7.x 기동 시 검증 실패 시 크래시. 유효한 LANGSMITH_API_KEY(LangGraph Cloud 접근 계정) 또는 LANGGRAPH_CLOUD_LICENSE_KEY 필요. (트러블슈팅 §12 참고.)

## 3. 검증 결과 (로그·모니터링)

### 3.1 AI 서버 로그

- **Postgres:** `Connecting to Postgres` → `postgres-service:5432/langgraph`, `Postgres checkpointer started with batching`, `Migration lock acquired/released`, `Postgres pool stats`.
- **Redis:** `Connected to Redis`, `Using Redis pub/sub for Core API`, `Redis pool stats` (idle_connections=1, max_connections=2000).
- **워커:** `Starting 40 background workers`, **Worker stats:** `active=0` / **available=40** / **max=40** (파드당 40개).
- **기동:** `Application startup complete`, `Uvicorn running on http://0.0.0.0:8000`.

### 3.2 워커 수 해석

- **파드당 40 워커**가 기동 중이며, 런은 Postgres 큐에 enqueue되고 여러 스레드(`ThreadPoolExecutor-1_0`, `1_1`, `1_2` 등)에서 처리됨.
- **AI 파드 2개**일 때 **총 80 워커**가 동시에 런을 처리할 수 있는 구조. (이전 inmem 시절의 “실질 1~3개”와 명확히 구분됨.)

### 3.3 Locust·모니터링 (참고)

- Locust: 약 **90.7 RPS** 도달, **약 24% 실패**. 95th percentile 초반 스파이크 후 안정 구간 존재.
- 모니터링: 요청 수 약 **40개** 구간에서 **성공률 급락·504 Gateway Timeout** 발생.
- **실제 AI 서버가 처리한 요청 수** = 성공한 요청 수 (Locust: Total requests × (1 − Failures %)).

---

## 4. 알려진 제약·원인 가설

- **24% 실패 / 504:** (1) **Gemini API 16 RPS** 등 외부 할당량 초과, (2) Ingress/프록시·AI 타임아웃, (3) 스케일링(파드/리소스) 부족 등이 복합적으로 작용할 수 있음.
- **그래프 노드:** 여전히 **동기 `def`** 위주이면, 노드 내부에서 GIL 영향으로 동시 실행이 제한될 수 있음. 40 워커는 “동시에 처리 가능한 런 개수”이고, 노드가 blocking이면 그만큼 활용이 떨어질 수 있음.

## 6. 요약 문장

- **빌드:** Jenkins에서 `langgraph build`로 AI 이미지 빌드 후 Harbor 푸시. K8s는 포트 8000, REDIS_URI·DATABASE_URI 적용.
- **워커:** 파드당 40개 기동 확인. 파드 2개 시 **총 80 워커**가 돌아가는 구조로, 이전 대비 확실히 증가.
- **한계:** 실패율·504는 외부 API 제한(예: 16 RPS)·타임아웃·스케일링 등과 함께 보며, “실제 처리된 요청”은 성공 건수로 집계하면 됨.
