**정의:** 이 문서가 **체크포인트 1**이다.  
**의미:** “체크포인트 1부터 작업해줘” / “체크포인트 1 불러와”라고 하면 **이 문서를 읽고 이 시점 상태를 기준으로 작업**하면 된다. 이전 대화 내용은 포함하지 않는다.

---

## 1. 이 시점에서의 결론

- **5단계 전략( LangGraph Production → Redis → N_JOBS 검증 → 메모리 → KEDA )은 가능한 전략이다.**
- **순서:** 1 → 2 → 3 → 4 → 5. 3번(노드 `async def` 전환)을 안 하면 N_JOBS=40을 줘도 GIL 때문에 1~3개만 실질 동작한다.

---

## 2. 체크포인트 1 시점의 5단계 상태 (한 줄 요약)

| 단계 | 항목 | 상태 (2026-02 업데이트) |
|------|------|-------------------------|
| 1 | LangGraph Production (dev 탈출) | ✅ **적용 완료** — `langgraph build` 이미지, Jenkinsfile 전환, 포트 8000 |
| 2 | Redis 배포·연결 | ✅ **적용 완료** — Redis Deployment/Service, REDIS_URI/DATABASE_URI(ai-secret) |
| 3 | N_JOBS=40 + async | **모든 그래프 노드 동기 `def`** → GIL로 1~3개만 동작 (미변경) |
| 4 | 메모리 | requests 2Gi, prod에서 limits 제거 → 40개 동시 시 모니터링 필요 |
| 5 | KEDA (큐 기반 HPA) | 미도입, CPU/메모리 HPA만 사용 |
