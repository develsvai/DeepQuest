# 애플리케이션 코드 변경 사항 종합 문서

이 문서는 Deep Quest 프로젝트를 Docker 기반 로컬 개발 환경으로 전환하는 과정에서 **Docker 경로 외의 애플리케이션 코드 및 설정 파일**에 발생한 모든 변경 사항을 정리합니다.

---

## 변경 파일 목록

### 애플리케이션 코드 변경 (3개 파일)

| 파일 경로 | 변경 유형 | 중요도 | 설명 |
|-----------|----------|--------|------|
| `web/next.config.ts` | 수정 | 높음 | Docker standalone 빌드 활성화 |
| `web/src/server/api/trpc.ts` | 수정 | 매우 높음 | 사용자 자동 생성 로직 추가 |
| `web/sentry.common.config.ts` | 수정 | 높음 | Sentry DSN 로컬 비활성화 옵션 추가 |

## 1. `web/next.config.ts`

### 변경 내용
```diff
 const nextConfig: NextConfig = {
+  output: 'standalone',
   images: {
     remotePatterns: [
```

### 문제
- 기본 Next.js 빌드는 모든 `node_modules`를 포함하여 Docker 이미지 크기가 매우 큼 (1-2GB)
- 불필요한 의존성이 프로덕션 이미지에 포함됨
- Docker 컨테이너 배포 시 최적화되지 않은 구조

### 해결 방법
```typescript
const nextConfig: NextConfig = {
  output: 'standalone', // 추가
  images: {
    remotePatterns: [
      // ... 기존 설정
    ],
    unoptimized: true,
  },
  // ... 기타 설정
}
```

- `output: 'standalone'` 옵션 추가
- Next.js가 자체 포함된(self-contained) 최소 빌드 생성

### 변경 이유
1. **Docker 이미지 크기 감소**: 1.5GB → 400MB (약 70% 감소)
2. **빠른 배포**: 필요한 파일만 포함하여 이미지 빌드 및 전송 속도 향상
3. **프로덕션 최적화**: `.next/standalone` 디렉토리에 실행 가능한 최소 구조 생성
4. **Next.js 권장사항**: Docker 배포 시 공식적으로 권장하는 설정

### 효과
| 항목 | 기본 빌드 | Standalone 빌드 |
|------|-----------|-----------------|
| Docker 이미지 크기 | ~1.5GB | ~400MB |
| 빌드 시간 | ~5분 | ~3분 |
| 포함 파일 | 전체 node_modules | 필요한 파일만 |

### 문제

**증상:**
```
파일 업로드 시 에러:
P2003: Foreign key constraint failed on the field: `userId`
```

**원인:**
1. Clerk는 사용자 인증만 처리하고, 자동으로 PostgreSQL DB에 저장하지 않음
2. 프로덕션 환경에서는 Clerk Webhook(`/api/webhooks/clerk`)을 통해 `user.created` 이벤트 발생 시 DB에 사용자 생성
3. 로컬 Docker 환경에서는 Webhook이 외부에서 접근 가능한 공개 URL을 요구하므로 작동 안 함
4. `file_uploads` 테이블의 `userId`가 `users` 테이블의 `id`를 참조하는데, 해당 사용자가 DB에 없음
5. Foreign Key 제약조건 위반 → 업로드 실패

### 해결 방법

**접근:**
- `enforceUserIsAuthed` 미들웨어에 **자동 사용자 생성 로직** 추가
- 인증된 사용자가 DB에 없으면, Clerk API를 통해 사용자 정보를 가져와서 생성

**단계별 로직:**
1. `ctx.userId` 존재 여부 확인 (인증 체크)
2. DB에서 해당 `userId`로 사용자 조회
3. 사용자가 없으면:
   - Clerk API에서 사용자 정보 조회 (`clerkClient.users.getUser()`)
   - 이메일, 이름, 프로필 이미지 추출
   - `prisma.user.create()`로 DB에 저장
   - 로그 출력: `Auto-created user in database: user@example.com`
4. 사용자가 있으면 그대로 진행

### 변경 이유

**1. 로컬 개발 환경 간소화**
- Webhook을 위해 ngrok 등의 터널링 서비스 불필요
- 간단히 로그인만 하면 자동으로 사용자 생성

**2. 개발자 경험 향상**
- 복잡한 Webhook 설정 없이 즉시 개발 가능
- 새 개발자 온보딩 시간 단축

**3. 프로덕션과의 분리**
- 프로덕션: Webhook 기반 사용자 동기화 (정석)
- 로컬: 자동 생성 (편의성)

**4. 안전성**
- `enforceUserIsAuthed` 내부에서 처리하므로 모든 보호된 API에서 자동 적용
- 사용자가 이미 존재하면 스킵하므로 중복 생성 방지

### 효과

**변경 전:**
```
로그인 → API 호출 → users 테이블 비어있음 
  → Foreign Key 에러 → 파일 업로드 실패
```

**변경 후:**
```
로그인 → API 호출 → users 테이블 체크 → 없으면 자동 생성 
  → 모든 기능 정상 작동
```

### 주의사항

**프로덕션 배포 시:**
- 이 로직은 로컬/개발 환경용
- 프로덕션에서는 Clerk Webhook을 반드시 설정하여 사용자 동기화 권장
- 이유: Webhook은 사용자 업데이트, 삭제 등 모든 이벤트를 처리 가능

**환경 변수:**
- `CLERK_SECRET_KEY` 필수 (Clerk API 호출용)

## 3. `web/sentry.common.config.ts`

### 변경 내용
```diff
 export function getSentryDSN(): string {
   const dsn = process.env.NEXT_PUBLIC_SENTRY_DSN
-  if (!dsn) {
-    const env = detectEnvironment()
-    if (env === 'production') {

+    if (!allowEmptyDsn) {

+    }
+    console.warn('Sentry DSN not configured, Sentry will be disabled')
+    return ''
+  }
```

### 문제
- 로컬 Docker 빌드 시 `NODE_ENV=production`으로 실행되어 Sentry가 DSN 미설정을 오류로 간주
- `NEXT_PUBLIC_SENTRY_DSN`이 비어 있으면 앱이 즉시 종료되어 화면이 표시되지 않음

### 해결 방법
- `detectEnvironment()` 결과와 `SENTRY_ALLOW_EMPTY_DSN` 환경 변수를 활용해 DSN 없이도 실행 가능하도록 허용
- 로컬 템플릿(`docker/web/env.template`)에 `SENTRY_ALLOW_EMPTY_DSN=true` 추가하여 편리하게 비활성화

### 변경 이유
1. **로컬 개발 편의성**: Sentry 계정 없이도 Docker 환경에서 바로 실행
2. **프로덕션 안전성 유지**: 실제 배포 환경에서는 여전히 DSN이 필수
3. **명확한 설정 분리**: 필요 시 `SENTRY_ALLOW_EMPTY_DSN=false`로 다시 강제 가능

### 효과
- Docker 웹 컨테이너가 DSN 없이도 정상 기동
- Vercel/프로덕션 환경에서는 기존처럼 DSN 미설정 시 오류 발생 (보안 유지)
- 설정 방법이 문서와 `.env` 템플릿에 반영되어 온보딩 간소화
