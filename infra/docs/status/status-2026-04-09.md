## 의도

## 현재 상태

- `kubectl`과 `ssh station` 실측 기준으로 `kubernetes-admin@kubernetes` 컨텍스트에서 `deep-quest`, `argocd`, `harbor`, `jenkins`, `monitoring`, `keda` 구성이 다시 확인됐다
- prod 배포 흐름은 여전히 Jenkins가 Harbor에 이미지를 push하고 `deploy` 브랜치의 `k8s/overlays/prod/kustomization.yaml` tag를 갱신하면 ArgoCD application `deepquest-infra`가 `k8s/overlays/prod`를 sync하는 구조로 유지된다
- prod 앱 런타임 기준 autoscaling은 web HPA와 ai KEDA / PDB 분리 구조이며, secret은 Kustomize base에서 제외되고 `k8s/scripts/apply-secrets.sh` 또는 직접 `kubectl apply` 흐름으로 관리된다
- NFS 동적 프로비저닝 backend는 `kube-system/nfs-provisioner` 기준 `192.168.0.7:/data/nfs`로 확인됐다

## 알려진 이슈

- Proxmox 계층 자체의 VM 생성, 스토리지, 네트워크, HA 정보는 현재 컨텍스트에서 직접 관측되지 않는다
- `ai-server-keda`는 trigger active 상태지만 `autoscaling.keda.sh/paused: true`여서 현재 AI 자동 확장 의도와 실운영 상태가 어긋나 있을 수 있다
- `nfs-storage`가 사실상 단일 backend `192.168.0.7:/data/nfs`에 의존하는지, failover / DR 정책이 있는지는 현재 문서에 드러나지 않는다
- worker `5`대 사용 이유와 node role separation 기준은 현재 활성 문서에 정리돼 있지 않다
