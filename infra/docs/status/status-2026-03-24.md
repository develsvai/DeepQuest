## 의도

- `deep-quest` AI 처리 경로를 backlog 기준으로 관측 가능한 상태로 만들고, autoscaling이 실제 처리량 증가로 이어지지 못하게 막는 병목을 단계적으로 제거한다
- `worker 포화 -> backlog 적체 -> scale-out -> 실제 처리량 증가` 흐름이 운영 지표로 설명 가능하도록 맞춘다

## 알려진 이슈

- PDF 전체를 base64로 LLM에 전달하는 구조는 여전히 요청당 처리시간을 키우는 후보로 남아 있다
- KEDA scale 신호와 실제 `ai-server` ready 사이에는 배포 및 초기화 지연이 남아 있다
- Postgres connection 수를 실험용으로 크게 올린 상태라 운영 적정값은 별도 재산정이 필요하다
- tail latency와 일부 `502`는 개선됐지만 완전히 제거되지는 않았다
- 현재까지의 개선이 운영 기본값인지 실험 중 임시 상향값인지 다시 구분할 필요가 있다
