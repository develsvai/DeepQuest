## 현재 상태

- `langgraph.run` 기반 metrics exporter와 관련 SQL 보정이 반영돼 queue backlog, worker busy, completed/sec, failed/sec, duration 지표를 계속 볼 수 있는 상태다
- `concurrent_max=40`과 Redis 기반 Gemini 전역 rate limiter가 적용된 상태에서 초기 부하 실험을 마쳤고, worker 포화와 backlog 적체 구조를 확인했다
- KEDA `ScaledObject`는 `ai_job_queue_pending` 기준으로 동작하며, `ai-server cpu request=1` 병목을 `200m` 조정으로 완화한 뒤 autoscaling이 실제 처리량 증가로 이어지는 것을 확인했다
- 개선 후 재실험에서는 `ai-server available=10`, `Worker Busy≈400`, `Jobs Completed/sec≈20`, `Queue Backlog 감소`까지 확인했다
- Postgres는 실험 중 `max_connections=100 -> 200 -> 500`까지 상향했고, async 전환 작업도 주요 AI 처리 경로에 반영된 상태다

## 알려진 이슈

- PDF 전체를 base64로 LLM에 전달하는 구조는 여전히 요청당 처리시간을 키우는 후보로 남아 있다
- KEDA scale 신호와 실제 `ai-server` ready 증가 사이에는 배포 및 초기화 지연이 남아 있다
- 실험 중 상향한 Postgres `max_connections=500`과 메모리 값은 운영 적정값 재산정이 필요하다
- 같은 조건에서 `20ops` 이상 처리량이 계속 재현되는지는 아직 다시 확인하지 않았다
- `pdf 분석 개선`은 현재 작업 범위에서 보류 상태다
