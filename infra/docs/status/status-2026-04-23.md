## 현재 상태

- 이전 성능 문서 축에는 K6 Web 계층, 초기 Locust soft failure, `concurrent_max=40` worker 포화, backlog 적체, production image/Redis/Postgres 전환까지 정리돼 있다
- raw evidence에는 `concurrent_max=40`, Gemini 전역 limiter, queue backlog metric, `ai-server-keda` scale-out, 처리율 개선 재실험 결과가 남아 있다
- evidence 기준으로 `Gemini 2.5 flash-lite` 호출은 클러스터 내부 limiter `60 RPS` 아래에서 통제됐고, `concurrent_max=40`에서는 worker 포화와 backlog 적체가 확인됐다
- 후속 KEDA 재실험에서는 `ai-server desired=10`, 실제 `available=10`, worker busy 약 `400`, `Jobs Completed/sec` 약 `20`, backlog 감소 전환이 관찰됐다
- 현재 메인 성능 문서는 이 후속 실험 결과를 아직 반영하지 않아, evidence와 canonical summary 사이에 간격이 있다

## 알려진 이슈

- Google quota를 `4000 RPM` 수준으로 올린 운영 변경과, 실제 클러스터 limiter `60 RPS` 실험값 사이의 관계가 메인 문서에 아직 명확히 구분돼 있지 않다
- 처리량은 `4 ops -> 20 ops` 수준으로 개선됐지만 `P95/P99` tail latency는 여전히 높게 남아 있다
- 현재 성능 문서에는 KEDA retry 단계별 결과와 최종 `400 workers` 해석이 부족하다
