## 의도

## 현재 상태

- 전날 작업 기준으로 metrics exporter, Redis 기반 Gemini rate limiter, KEDA `ScaledObject`, `N_JOBS_PER_WORKER=40`, Postgres `max_connections` 상향, async 전환이 모두 반영된 흐름은 유지되고 있다
- 같은 실험 조건에서 `20ops` 이상 처리량이 다시 재현되는 것까지 확인돼, 처리량 개선 흐름은 일회성 결과로만 보지 않는 상태다
- prod 배포의 실제 반영 경로는 Jenkins 빌드, Harbor push, `deploy` 브랜치 overlay 태그 갱신, ArgoCD 수동 sync 순서로 이어지는 구조이며, 부분 빌드 시 태그 정합성 확인이 필요한 상태다
- 남은 핵심 쟁점은 운영 기준 Postgres 값 재산정, KEDA scale 신호와 실제 ready 증가 사이 지연 확인, PDF 전체 base64 전달 구조 최적화다

## 알려진 이슈

- Postgres `max_connections=500`과 메모리 상향값은 실험 기준으로는 유효했지만, 운영 기본값으로 유지할 적정 수준은 아직 확정하지 못했다
- KEDA scale 신호와 실제 `ai-server` ready 증가 사이 지연은 여전히 운영 관점에서 따로 확인이 필요하다
- PDF 전체를 base64로 전달하는 구조는 처리시간과 비용 측면에서 계속 병목 후보로 남아 있다
- `pdf 분석 개선`은 현재 작업 범위에서 보류 상태다
