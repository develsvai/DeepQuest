# 작업 계획

날짜: `2026-04-23`

## 계획

현재 상태 요약: 최신 기준 상태는 [status-2026-04-23.md](../status/status-2026-04-23.md)에 있고, 오늘 작업은 AI rate limiter/KEDA evidence를 메인 성능 문서로 승격하는 데 집중한다.

1. evidence를 읽고 단계별 결과를 다시 묶는다
   - `concurrent_max=40` evidence에서 worker 포화, backlog 적체, limiter 값을 다시 확인한다
   - `keda` evidence에서 `desired/available replicas`, worker 수, backlog 변화, completed/sec 변화를 단계별로 비교한다
2. 메인 성능 문서를 갱신한다
   - 후속 AI scaling 실험을 별도 성능 요약 문서로 만든다
   - `docs/performance/README.md`와 관련 문서를 새 요약 기준으로 연결한다
   - 필요하면 `capacity-plan-1000-rpm.md`의 검증된 사실을 최신 실험 결과 기준으로 보강한다
3. 오늘 반영한 범위와 남은 리스크를 기록한다
   - 오늘 문서화 범위를 `task`에 남긴다
   - 아직 evidence에만 남아 있는 추가 실험이나 남은 질문을 `보류 / 다음 액션`에 정리한다

## 진행

- 시작 전: `docs/performance/` 메인 문서에는 soft failure, `concurrent_max=40`, 초기 Locust/K6 요약은 있지만, 후속 KEDA retry와 `20 ops/s` 개선 결과는 아직 독립 문서로 승격되지 않은 상태였다
- evidence를 다시 읽어 아래 사실을 재확인했다
  - `concurrent_max=40` 단계에서 worker `active=40 available=0 max=40`, backlog 적체, `Jobs Completed/sec` 약 `0~1.2/s`, Gemini limiter `60 RPS`가 확인됐다
  - `KEDA + Postgres 200` 재실험에서는 `desired=10`, 실제 `available=4`, worker busy 약 `160`, `Jobs Completed/sec` 약 `1.8~2.0/s`, backlog `18K` 증가, `Insufficient cpu`가 확인됐다
  - `KEDA + cpu request 200m + Postgres 500 + async 개선` 재실험에서는 `desired=10`, `available=10`, worker busy 약 `400`, backlog `10K -> 7K`, `Jobs Completed/sec` 약 `3~4 -> 20`, aggregate `131.81 req/s`, 실패율 `0.04%`가 확인됐다
- 위 evidence를 기준으로 메인 성능 문서를 갱신했다
  - [ai-rate-limit-and-keda-summary.md](../performance/ai-rate-limit-and-keda-summary.md)를 새로 추가했다
  - [README.md](../performance/README.md)에 새 요약 문서와 실험 순서를 반영했다
  - [locust-day2-summary.md](../performance/locust-day2-summary.md)에 후속 실험 문맥 링크를 추가했다
  - [capacity-plan-1000-rpm.md](../performance/capacity-plan-1000-rpm.md)에 최신 검증 사실과 남은 병목을 보강했다
