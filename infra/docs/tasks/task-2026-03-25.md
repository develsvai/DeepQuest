# 작업 계획

날짜: `2026-03-25`

## 계획

현재 상태 요약: 전날 작업으로 metrics exporter, rate limiter, KEDA, AI worker concurrency, Postgres 상향, async 전환, `20ops` 수준 처리량 개선까지 확인됐다. 오늘 작업은 이 결과를 운영 기준으로 다시 좁히고, 남은 병목 후보를 다음 작업 단위로 분리하는 데 집중한다.

1. 운영 기준 Postgres connection 및 memory 값을 다시 산정한다
   - 실험 중 상향한 `max_connections=500`과 메모리 설정이 운영 기본값으로 유지 가능한지 확인 대상을 정리한다
   - connection ceiling, 실제 부하 시 사용량, 이후 저장소 반영 필요 여부를 판단할 기준을 묶는다
2. `ai-server` request와 autoscaling 기준을 운영 관점으로 다시 확인한다
   - `ai-server cpu request=200m` 조정 이후 replica 증가가 실제 운영 기본값으로 적절한지 검토 대상을 정리한다
   - KEDA scale 신호와 실제 ready 증가 사이 지연을 다시 볼 체크포인트를 남긴다
3. PDF 전달 방식과 추가 async 최적화 범위를 후속 작업으로 정리한다
   - PDF 전체 base64 전달 구조가 여전히 주요 병목 후보인지 확인 대상을 구체화한다
   - 이미 async 전환한 경로와 추가로 손봐야 할 경로를 분리한다
4. 같은 조건에서 `20ops` 이상 처리량이 재현되는지 다시 검증할 준비를 한다
   - baseline 복구, queue 초기화, limiter 값, worker concurrency, replica 수를 같은 조건으로 다시 맞춘다
   - 처리량 재현 시 같이 볼 backlog, worker busy, completed/sec 지표를 정리한다

## 진행

- 전날 작업 결과를 기준으로 현재 운영에서 다시 확인해야 할 항목을 정리했다.
  - metrics exporter, Redis 기반 Gemini rate limiter, KEDA `ScaledObject`, `N_JOBS_PER_WORKER=40`, Postgres `max_connections` 상향, async 전환, `20ops` 처리량 개선이 이미 반영된 작업 축으로 확인된다.
  - 남은 핵심 쟁점은 운영 기준 Postgres 값 재산정, KEDA ready 지연, PDF 전달 구조 최적화, 같은 조건의 처리량 재현 여부다.
- 4번 작업 기준으로 같은 실험 조건에서 `20ops` 이상 처리량이 다시 재현되는 것을 확인했다.
  - baseline 복구, queue 초기화, limiter 값, worker concurrency를 맞춘 뒤 처리량 재현 여부를 다시 확인한 결과, 전날 개선 흐름이 일회성은 아닌 것으로 판단했다.
