# 작업 계획

날짜: `2026-03-27`

## 계획

현재 상태 요약: 최신 기준 상태는 [status-2026-03-27.md](../status/status-2026-03-27.md)에 있고, 오늘 작업은 prod 부분 빌드 검증, Postgres `max_connections` 실측 경로 확인, `200ops` 병목 점검 순서 정리에 집중한다.

1. Jenkins 부분 빌드 수정이 실제 배포 흐름에서 의도대로 동작하는지 검증한다
   - [Jenkinsfile](../../Jenkinsfile) 의 재태깅 로직과 `deploy` 브랜치 overlay 갱신 로직을 다시 짧게 점검한다
   - 실제 실행 가능한 Jenkins job 파라미터와 검증 기준을 정리한다
   - 검증 시 확인 대상은 Harbor의 `${DEPLOY_ENV}` / `${BUILD_TAG}` 태그, `deploy` 브랜치의 [k8s/overlays/prod/kustomization.yaml](../../k8s/overlays/prod/kustomization.yaml), ArgoCD 반영 전후 이미지 정합성으로 둔다
2. Postgres `500` 메모의 실제 출처를 live state 기준으로 확정한다
   - 현재 저장소 기준 설정 경로와 live cluster의 실제 설정 경로가 일치하는지 확인한다
   - 확인 대상은 `show config_file;`, `show max_connections;`, Postgres 컨테이너 내부 `/etc/postgresql/postgresql.conf`, 관련 pod spec 또는 image tag다
   - 결과에 따라 `수동 변경`, `미반영 이미지`, `기록 오차` 중 어느 쪽인지 판단 근거를 남긴다
3. `200ops` 병목 점검 순서를 같은 시점 기준 조회 항목으로 다시 묶는다
   - limiter, worker concurrency, queue backlog, DB connection ceiling, HPA, ingress readiness 및 메트릭을 한 묶음으로 정리한다
   - 기존 `20ops` 근처 baseline과 비교할 때 무엇을 같은 시점에 다시 봐야 하는지 체크리스트를 만든다
   - 인프라 설정만으로 설명 가능한 병목과 앱 런타임 추가 분석이 필요한 병목을 구분한다
4. 필요하면 상태 추적 문서를 함께 갱신할 준비를 한다
   - live state를 새로 수집하면 [2026-03-27-live.md](../infra_state/2026-03-27-live.md) 와의 차이를 확인한다
   - 단순 실행 로그는 이 파일 `진행`에 남기고, 재발 가능성이 있는 이슈만 troubleshooting 반영 후보로 둔다
