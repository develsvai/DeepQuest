# 작업 계획

날짜: `2026-03-26`

## 계획

현재 상태 요약: 전날까지 metrics exporter, rate limiter, KEDA, `N_JOBS_PER_WORKER=40`, Postgres 상향, async 전환, `20ops` 처리량 재현까지 확인됐다. 오늘 작업은 남은 세 축인 Postgres 운영값 재산정, KEDA ready 지연 확인, PDF 전달 구조 최적화 범위 정리에 집중한다.

1. prod 빌드 태그 태깅과 배포 반영 흐름을 먼저 점검한다
   - `Jenkinsfile`, `k8s/overlays/prod/kustomization.yaml`, `k8s/argocd/application-prod.yaml` 기준으로 이미지 빌드, Harbor push, `deploy` 브랜치 태그 갱신, ArgoCD 반영 순서를 확인한다
   - 부분 빌드 시 이미지 태그가 서로 다른 커밋으로 섞일 가능성이 있는지 확인 대상으로 둔다
2. Postgres `max_connections=500`과 메모리 상향값을 운영 기준으로 다시 산정한다
   - 실험 기준 상향값이 현재 운영 기본값으로 유지 가능한지 판단할 기준을 정리한다
   - 실제 connection ceiling, 부하 시 사용량, 저장소 반영 필요 여부를 확인 대상으로 둔다
3. KEDA scale 신호와 실제 `ai-server` ready 시간 사이 지연을 운영 기준으로 다시 확인한다
   - scale-out 신호가 실제 available replica 증가로 이어지는 데 걸리는 구간을 확인 대상으로 둔다
   - `ai-server cpu request=200m` 조정 이후에도 남는 지연 요인이 무엇인지 나눠 본다
4. PDF 전체 base64 전달 구조와 추가 async 최적화 범위를 후속 작업 단위로 정리한다
   - PDF 전달 방식이 처리시간과 비용에 주는 영향을 다시 볼 체크포인트를 만든다
   - 이미 async 전환된 경로와 추가 최적화가 필요한 경로를 구분한다
5. 위 항목들의 결과를 다음 병목 분석 기준으로 다시 묶는다
   - 운영 기본값으로 확정할 항목
   - 실험값으로만 유지할 항목
   - 앱 런타임 후속 분석으로 넘길 항목

## 진행

- 1번 작업 기준으로 prod 빌드 태그 태깅과 배포 반영 흐름을 저장소 기준으로 확인했다.
  - Jenkins가 선택한 이미지를 Harbor에 push하고 `deploy` 브랜치의 prod overlay 태그를 갱신한 뒤, ArgoCD가 이를 읽어 수동 sync로 반영하는 구조를 확인했다.
  - 부분 빌드 시 태그 혼합 위험이 있어 이후 빌드하지 않은 이미지도 새 `${BUILD_TAG}`로 재태깅하고 overlay 세 이미지 태그를 함께 갱신하는 방향으로 수정되었다.
