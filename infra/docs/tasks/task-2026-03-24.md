# 작업 계획

날짜: `2026-03-24`

확인 시점: `2026-03-24 ~ 2026-03-25`

대상 환경: `deep-quest prod`, `monitoring`, `KEDA`, `LangGraph run metrics`

## 계획

1. 관측 체계를 구축한다
   - `langgraph.run` 기반 metrics exporter를 구현해 queue backlog, worker busy, completed/sec, failed/sec, success rate를 수집 가능하게 만든다
   - exporter와 SQL 오류를 수정해 메트릭 수집이 실제로 끊기지 않도록 정리한다
2. AI 부하 실험 준비를 마친다
   - `concurrent_max=40` 설정의 실제 적용 상태를 확인한다
   - Locust 시나리오를 `enqueue -> backlog 관찰` 목적에 맞게 수정한다
   - baseline 복구, 큐 초기화, 로그와 캡처 수집 절차를 정리한다
3. 전역 유량 제어를 적용한다
   - Redis 기반 Gemini 전역 rate limiter를 도입한다
   - `RPS`, `burst`, `wait timeout`을 조정하고 실험 중 동작 여부를 검증한다
4. 1차 실험으로 `concurrent_max=40` 동작을 검증한다
   - worker 포화와 backlog 적체 여부를 확인한다
   - 즉시 장애가 아니라 queue 적체 구조로 버티는지 확인한다
5. KEDA를 도입한다
   - KEDA 설치와 `ScaledObject` 적용을 완료한다
   - `ai_job_queue_pending` 기반 autoscaling 구조로 전환한다
   - 기존 HPA 충돌과 PDB 문제를 함께 정리한다
6. KEDA 초기 병목을 분석한다
   - desired replica와 available replica의 차이를 확인한다
   - `ai-server cpu request`, 클러스터 CPU 예약, Postgres connection ceiling을 병목 후보로 검증한다
7. 인프라 튜닝을 적용한다
   - `ai-server cpu request`, Postgres `max_connections`, Postgres 메모리 설정을 조정한다
8. 애플리케이션 비동기화를 적용한다
   - `resume_parser`, `question_feedback_gen`, `question_gen`, `jd_structuring`, `jd_to_text` 경로를 async 중심으로 개선한다
   - `.invoke()`와 동기 stream 루프를 정리한다
9. 개선 후 재실험으로 처리량 개선 여부를 검증한다
   - available pod 증가, worker busy 증가, completed/sec 상승, backlog 감소 전환을 확인한다

## 진행

1. 관측 체계를 구축했다
   - `langgraph.run` 기반 metrics exporter를 붙여 `ai_job_queue_pending`, `ai_job_queue_processing`, `ai_worker_busy`, `ai_jobs_completed_total`, `ai_jobs_failed_total`, `ai_job_duration_seconds`를 수집하도록 정리했다
   - duration 대상 데이터가 0건일 때 빈 `HistogramMetricFamily`가 생성되며 죽는 문제를 보완했다
   - Postgres `FILTER` 문법 오류를 수정해 메트릭 수집이 실제로 되도록 만들었다
   - `Worker Busy`, `Queue Backlog`, `Jobs Completed/sec`, `Jobs Failed/sec`, `Worker Success Rate`를 기준으로 실험을 해석할 수 있는 상태를 만들었다
2. AI 부하 실험 준비를 마쳤다
   - `concurrent_max=40` 설정이 이미 적용돼 있음을 클러스터에서 확인했다
   - `N_JOBS_PER_WORKER=40`
   - `Worker stats ... max=40`
   - baseline 복구, queue 초기화, 단계별 캡처와 로그 수집 절차를 정리했다
3. 전역 유량 제어를 적용했다
   - Redis 기반 전역 Gemini rate limiter를 적용했다
   - `GEMINI_GLOBAL_RATE_LIMIT_RPS=60`
   - `GEMINI_GLOBAL_RATE_LIMIT_BURST=60`
   - `GEMINI_GLOBAL_RATE_LIMIT_WAIT_TIMEOUT_SECONDS=15`
   - 실험 중 Gemini 호출량이 제한 범위 근처로 모이도록 제어했다
4. 1차 실험으로 `concurrent_max=40` 동작을 검증했다
   - 100 VU 기준 초기 실험을 수행했다
   - worker 포화와 queue backlog 증가를 확인했다
   - 즉시 장애보다 backlog 적체로 시스템이 버티는 구조를 확인했다
5. KEDA를 도입했다
   - `ai_job_queue_pending` 기준 KEDA `ScaledObject`를 적용했다
   - KEDA 컨트롤러를 설치했다
   - 기존 autoscaling 충돌 요소와 PDB 구성을 함께 정리했다
6. KEDA 초기 병목을 분석했다
   - autoscaling은 동작하지만 실제 pod 증가가 제한되는 현상을 확인했다
   - `desired replica=10`까지 갔지만 `available replica`는 낮게 유지됐다
   - 원인은 `ai-server cpu request=1`과 클러스터 CPU request 예약 부족으로 확인됐다
   - Postgres `max_connections=100`에서 `too many clients already`가 발생해 DB connection ceiling도 병목으로 확인했다
7. 인프라 튜닝을 반영했다
   - `ai-server cpu request = 1 -> 200m`
   - Postgres `max_connections = 100 -> 200 -> 500`
   - Postgres 메모리 limit 상향 방향을 정리했다
8. 애플리케이션 병목을 줄이기 위해 비동기화를 적용했다
   - `resume_parser`: `requests.get + retry + time.sleep` 경로를 `httpx.AsyncClient` 기반 async로 전환했다
   - `.invoke()` 기반 LLM 호출을 가능한 경로에서 `.ainvoke()`로 전환했다
   - `question_feedback_gen`, `question_gen`, `jd_structuring`, `jd_to_text`를 async로 정리했다
9. 개선 후 재실험으로 처리량 개선을 검증했다
   - 각 실험 전후에 `run/thread/checkpoint` 계열 테이블을 비우고 queue를 초기화했다
   - `ai-server`를 `1/1`로 되돌리고 KEDA pause/unpause로 baseline을 맞췄다
   - 개선 후 재실험에서 autoscaling이 실제 처리량 증가로 연결되는 것을 확인했다
   - `ai-server available`: `4 -> 10`
   - `Worker Busy`: `160 -> 400`
   - `Jobs Completed/sec`: `4 -> 20`
   - `Queue Backlog`: 증가 상태에서 감소 상태로 전환
