# 작업 계획

날짜: `2026-04-09`

## 계획

현재 상태 요약: 최신 기준 상태는 [status-2026-04-09.md](../status/status-2026-04-09.md)에 있고, 오늘 작업은 prod 클러스터 구조를 실측 기준으로 정리하고 운영 리뷰 질문 목록까지 남기는 데 집중한다.

1. live cluster와 `station` 호스트를 조회해 현재 구조를 사실 기준으로 다시 묶는다
   - `kubectl`로 nodes, workloads, services, storage, autoscaling, RBAC, namespace 구성을 확인한다
   - `ssh station`으로 NFS backend와 호스트 기본 정보를 확인한다
   - `docs/infra_state/2026-03-27-live.md`와 비교해 현재 달라진 지점을 추린다
2. 저장소 매니페스트와 운영 문서를 대조해 구조 설명에 필요한 근거를 정리한다
   - `Jenkinsfile`, `k8s/argocd/application-prod.yaml`, `k8s/base/*`, `k8s/overlays/prod/*`를 기준으로 CI/CD, autoscaling, RBAC, secret 흐름을 확인한다
   - 실측으로는 안 나오는 항목은 추정하지 않고 확인 질문으로 남긴다
3. 결과를 문서로 남긴다
   - `docs/infra_state/2026-04-09-live.md`에 현재 상태 스냅샷을 남긴다
   - 클러스터 구조와 예상 질문을 `docs/reference/` 문서로 정리한다
   - 오늘 작업 축과 후속 액션을 이 `task`에 반영한다

## 진행

- 1번 작업 기준으로 live cluster와 `station` 호스트를 조회했다
  - `kubectl` 실측 기준으로 `deep-quest`, `argocd`, `harbor`, `jenkins`, `monitoring`, `keda` namespace와 master `1` / worker `5` 노드 구성이 확인됐다
  - `nfs-storage` backend는 `kube-system/nfs-provisioner` 기준 `192.168.0.7:/data/nfs`였고, `station`은 `jenkins-build` / `192.168.0.7`로 확인됐다
  - `deep-quest` 현재 workload, web HPA, ai KEDA / PDB, PVC, secret inventory, scheduled job 상태를 같은 시점 기준으로 다시 수집했다
- 2번 작업 기준으로 저장소 매니페스트와 운영 문서를 대조했다
  - Jenkins -> Harbor -> `deploy` 브랜치 overlay tag 갱신 -> ArgoCD sync 흐름이 현재 저장소 기준과 live cluster 기준에서 다시 일치했다
  - web은 CPU / Memory HPA, ai는 prod overlay에서 base HPA를 삭제하고 Prometheus queue metric 기반 KEDA로 전환한 구조임을 확인했다
  - `ai-server-keda`는 현재 `Active=True`지만 `Paused=True` 상태라 follow-up 질문으로 남겨야 한다는 점이 드러났다
  - secret은 base Kustomization에서 제외되어 있고 `apply-secrets.sh` 또는 직접 적용 흐름으로 관리된다는 점을 다시 확인했다
- 3번 작업 기준으로 문서화를 진행했다
  - [2026-04-09-live.md](../infra_state/2026-04-09-live.md)에 현재 상태 스냅샷을 남겼다
  - [prod-cluster-structure-and-review-questions.md](../reference/prod-cluster-structure-and-review-questions.md)에 구조 요약과 예상 질문 목록을 정리했다

## 보류 / 다음 액션

1. Proxmox 계층에서 master / worker / `jenkins-build`가 어떻게 VM과 스토리지로 매핑되는지 직접 확인한다
2. `ai-server-keda`가 왜 pause 상태인지, 일시 조치인지 현재 운영 정책인지 확인한다
3. `192.168.0.7:/data/nfs` 장애 시 영향 범위와 backup / DR 절차를 확인한다
4. worker `5`대 구성 이유와 node role separation 기준을 운영 문서에 승격할지 판단한다
5. `verify-deploy.sh`가 prod 배포 검증 시 현재 checkout의 placeholder tag가 아니라 `deploy` 브랜치 overlay의 `${BUILD_TAG}` 기준으로 live image를 비교할 수 있게 개선한다
   - 작업자 관점에서는 `deploy` 브랜치 checkout이나 `KUSTOMIZATION_FILE` 수동 지정 없이도 release 정합성을 검증할 수 있어야 한다
   - 후보 형태: `VERIFY_OVERLAY_REF=deploy EXPECTED_TAG=<BUILD_TAG> scripts/verify/verify-deploy.sh` 또는 별도 `scripts/verify/verify-release.sh <BUILD_TAG>`
