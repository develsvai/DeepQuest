# 서비스 핵심 철학

## 개발의 대원칙

### 설계 철학: Start with Why

Simon Sinek의 **"Start with Why"** (2009) 개념을 기반으로 원칙을 세운다.

**Golden Circle 적용:**

```
WHY  → 왜 존재하는가? (목적, 신념)
HOW  → 어떻게 하는가? (차별화된 방법)
WHAT → 무엇을 하는가? (제품, 서비스)
```

**핵심 원칙:**

- **Why부터 시작**: 기능(What) 나열이 아닌, 서비스의 존재 이유와 신념을 먼저 전달
- **감정에 호소**: 사람들은 "무엇"을 사는 게 아니라 "왜"를 산다
- **공감 → 솔루션 순서**: 문제에 공감한 후 해결책 제시

### 설계 원칙

#### 내가 믿는 'WHY'가 서비스 전반에 배치되어야 한다

- '실력 있는 개발자' 인재상에 대한 믿음(면접에서는 이러한 것을 물어본다)
  - 자신의 경험에 대해 깊이 있게 설명 가능하다
    - '깊이 있다'라는 것은,
      - '기술적 의사결정 / 기술적 깊이와 원리 / 문제 해결 및 검증 / 확장 가능성' 등의 4가지 카테고리에 답할 수 있다는 뜻
  - 그러한 설명을 단순히 외우는 것이 아니라, 자신만의 언어로 표현 가능하다
    - '자신만의 언어'란 무엇인지에 대한 추가 고민 필요
      - 자신만의 비유법인지? 화법인지?
      - 확실한 것은 말로써 연습해야 함. 텍스트는 입력이 불가능한 상황일 때만
- 기술면접은, 지원자가 '실력 있는 개발자'인지를 검증하는 토론이다
  - 시험문제(trivial questions)를 푼다기 보단, 기술적 주제에 대한 대화, 토론에 가깝다
- 구직자들은, 이러한 '실력 있는 개발자'가 되고 싶어하고, 기업은 이러한 개발자 찾고 싶어 한다
  - 하지만 둘 다 어떻게 해야 '실력 있는 개발자'가 될 수 있는지, 검증할 수 있는지 그 방법을 모른다
  - 알아도 너무 비싸거나 쉽지 않다
- AI가 발전할 수록, 이러한 '실력 있는 개발자'에 대한 수요는 늘어날 것이다
  - AI의 발전으로, 겉으로 '그럴듯해'보이는 개발자들이 늘어났다
    - 기술이 발전할 수록, 인간만이 할 수 있는 역량이 중요해진다
      - AI의 역량: 구현
      - 인간의 역량: 의사결정(무엇을, 어떻게, 왜 하는지)
    - 이러한 인간만의 역량은 기존의 시험방식(코테, 과제 등)으로는 검증하기 어렵다. 결국 대면면접이 더욱 중요해질 것이다

#### "WOW" Point(사용자의 일반적인 기대를 넘는 포인트)가 많아야 한다

- ui/ux로의 wow point
  - 최소 input, 높은 퀄리티의 output을 빠르게 경험
    - e.g., 이력서만 넣었는데, 내 경험을 이렇게 체계적으로 분석해주다니!
    - 고려할 한계:
      - 개인화의 깊이:
        - 이력서 분석이 '체계적'이긴 하지만, 사용자가 이미 알고 있는 사실을 단순히 요약한 수준이라면 감탄은 금방 식을 수 있습니다.
        - 타겟 유저(서류는 통과하지만 면접에서 떨어지는 경력 개발자)의 pain point는 이력서 자체가 아닌, **면접에서 경험을 깊이 있게 설명하지 못하는 것**입니다.
        - Wow Point 예시 방향 (추가 기획 필요):
          - **면접 공격 포인트 예측**: "이 프로젝트에서 Redis 대신 왜 Kafka를 선택하셨나요?" — 이 질문이 나올 확률 높음. 현재 이력서에는 '왜'가 드러나지 않습니다.
          - **깊이 표현 기회 발굴**: "이 경험에서 '성능 30% 개선'이라고 적으셨는데, 어떤 병목을 발견하고 어떻게 측정했는지를 설명하면 기술적 깊이를 보여줄 수 있습니다."
          - **당연시하는 것의 가치 재발견**: "MSA 전환 과정에서 서비스 경계를 어떻게 나눴는지 — 이건 의사결정 역량을 보여주는 핵심 포인트인데, 지금은 '전환했다'로만 끝나 있어요."
      - 대기 시간의 병목:
        - '빠르게' 경험하는 것이 중요하지만, 복잡한 분석은 AI 처리 시간이 필요합니다.
        - 이 '기다리는 시간'마저 인터랙티브한 경험으로 채워야 Wow Point가 훼손되지 않습니다.

#### Devide and Conquer 전략을 적용한다

- '서비스 성공'이라는 모호한 목표 설정 말고, 수치로 증명할 수 있는 가장 작은 목표(핵심지표)부터 달성을 목표로 한다
  - e.g., 26년 1월의 목표는 이력서 분석 결과 10개 달성. 2월의 목표는 문제 생성 경험 10개 달성, ...
- 아주 작은 타겟 고객의 문제를 해결하는 것을 목표로 한다
  - 현재 단계에서 타겟 고객 가설은 검증이 필요하다
    - 내가 생각했을 때의 타겟 고객
      - 이직을 시도하고 있어 서류에는 합격하지만, 기술면접에서 떨어지고 있는 2년차 이상의 경력 개발자
