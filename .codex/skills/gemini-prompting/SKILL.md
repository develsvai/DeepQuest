# Gemini 프롬프트 설계

## 목표

`ai/`의 Gemini 기반 프롬프트를 현재 그래프 구조, 상태 스키마, 출력 계약에 맞게 안정적으로 설계한다.
핵심은 "그럴듯한 문장"보다 "현재 노드가 실제로 소비할 수 있는 출력"을 만드는 것이다.

## 언제 쓰나

- `ai/src/graphs/**/prompts.py`를 수정할 때
- Gemini 응답 품질이 흔들리거나 형식 오류가 반복될 때
- 기존 OpenAI/Anthropic 중심 프롬프트를 Gemini 쪽으로 옮길 때
- 질문 생성, 피드백 생성, JD 구조화, 이력서 파싱처럼 구조화 출력이 중요할 때

## 작업 순서

1. 먼저 아래를 확인한다
   - `ai/README.md`
   - 관련 그래프의 `state.py`, `schema.py`, `nodes.py`, `prompts.py`
   - 관련 `ai/docs/*.md`
2. 프롬프트가 만족해야 하는 출력 계약을 먼저 적는다
   - 필수 필드
   - 금지 필드
   - 길이 제약
   - 언어 제약
   - 후속 노드가 기대하는 형식
3. 현재 프롬프트를 아래 기준으로 분해한다
   - 역할
   - 입력 문맥
   - 제약
   - 출력 형식
   - 예시
4. 형식 안정성이 중요하면 `references/gemini-checklist.md`의 구조를 따른다
5. 프롬프트 변경 후에는 가능한 한 실제 샘플 입력으로 노드 단위 검증 계획을 함께 남긴다

## 규칙

- Gemini가 추론해야 할 맥락을 코드 밖에서 암묵적으로 기대하지 않는다
- 출력 형식이 중요하면 자연어 설명만 두지 말고 필드, 순서, 금지사항을 명시한다
- few-shot 예시는 형식이 흔들릴 때만 최소 개수로 넣고, 예시마다 포맷을 완전히 동일하게 유지한다
- "추측하지 마라" 같은 넓은 부정문보다 "제공된 입력만 사용하라"처럼 허용 범위를 긍정형으로 고정한다
- temperature, top-p 같은 파라미터는 프롬프트 안에서 상상하지 말고 현재 코드 설정을 먼저 따른다
- 프롬프트 수정이 상태 스키마나 후속 파서 가정까지 바꾸면 프롬프트만 고치지 말고 계약 변경으로 취급한다
- 긴 배경 설명은 프롬프트 본문에 늘어놓기보다 실제 입력 컨텍스트를 줄이는 쪽을 먼저 검토한다
