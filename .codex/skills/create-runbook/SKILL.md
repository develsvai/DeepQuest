# 런북 생성

## 목표

대화나 작업 기록에 남아 있는 실제 실행 절차를 `docs(haru2)/runbooks/` 아래의 재사용 가능한 runbook 문서로 정리한다.
핵심은 "무슨 일이 있었는지"가 아니라 "다음에도 같은 작업을 어떻게 재현할지"를 남기는 것이다.

## 작업 순서

1. `docs(haru2)/reference/document-writing-rules.md`를 읽는다
2. `docs(haru2)/runbooks/README.md`를 읽는다
3. 사용자가 지정한 채팅, 작업 기록, 관련 문서를 확인한다
4. 아래를 분리한다
   - runbook에 남길 반복 가능한 절차
   - troubleshooting으로 보내야 할 원인 분석, 우회 배경, 일회성 판단
5. 기존 `docs(haru2)/runbooks/`에 같은 절차 문서가 있으면 새 파일 대신 그 문서를 보강한다
6. 없으면 작업 성격이 드러나는 파일명으로 새 runbook 문서를 만든다
7. 문서에는 최소한 아래 내용을 정리한다
   - 상단의 `확인 시점`, `대상 환경`
   - 목적 또는 적용 범위
   - 사전 확인 사항
   - 실제 실행 순서
   - 검증 방법
   - 주의 사항 또는 되돌리기 조건

## 규칙

- 실제로 수행했거나 검증된 절차만 적는다
- 명령, 경로, 리소스 이름은 채팅과 레포 기준 실제 값으로 유지한다
- Markdown 링크 대상은 작성 중인 runbook 파일 위치 기준 상대경로로 쓴다
- 같은 디렉터리 문서는 `file.md`, 다른 문서 디렉터리는 `../troubleshooting/file.md`, 레포 루트 자산은 `../../web/path.ts`, `../../ai/path.py`처럼 작성한다
- `/Users/...`, `/home/...`, `file://...` 같은 로컬 절대경로를 링크 대상으로 쓰지 않는다
- 상단에 `확인 시점`과 `대상 환경`을 항상 넣는다
- 섹션 순서는 가능하면 `목적 -> 사전 확인 사항 -> 실행 순서 -> 검증 방법 -> 주의 사항`을 기본으로 따른다
- 현재 `docs(haru2)/runbooks/`의 문체처럼 간결한 제목과 짧은 설명 위주로 정리한다
- 배경 설명이 길어지면 runbook 본문에 늘어놓지 말고 필요한 최소 맥락만 남긴다
- 원인 분석, 장애 서사, 재발 분석은 `docs(haru2)/troubleshooting/`로 보낸다
- 비교표나 설계 배경은 `docs(haru2)/reference/`로 보낸다
- 일회성 상태 기록은 `docs(haru2)/status/` 또는 `docs(haru2)/tasks/`로 보낸다
- 기존 runbook에 자연스럽게 합쳐지면 새 파일을 만들지 않는다
- 기존 runbook이 여러 독립 절차를 품고 길어지고 있으면 새 절차는 별도 runbook으로 만든다
- 하나의 runbook은 하나의 반복 가능한 애플리케이션 작업 목표만 다룬다
- 예: 로컬 개발 환경 부트스트랩, `web/` 배포 전 점검, `ai/` 워커 실행 검증은 서로 다른 runbook 후보로 본다
- 150~200줄을 넘는 runbook에 새 절차를 붙일 때는 index 문서로 유지할지, 세부 runbook으로 분리할지 먼저 판단한다
- 문서 제목과 섹션은 현재 `docs(haru2)/runbooks/` 톤에 맞춰 간결하게 유지한다
