# 루트 애플리케이션 레이어 확장안

현재 `infra` 적용안을 루트에 그대로 복사하는 것보다, 애플리케이션 레이어 특성에 맞게 한 단계 확장하는 편이 더 좋다.

## 왜 그대로 복사하면 아쉬운가

`infra`는 사실 기반 상태, 명령 결과, 운영 스냅샷이 강한 레이어라 `bkit`을 보수적으로 붙이는 것이 맞다.
반면 루트는 `web/`와 `ai/`를 함께 조율하는 애플리케이션 레이어라, 코드 변경 맥락과 작업 설명을 더 많이 다룬다.

즉 루트는 단순 문서 초안기보다 "작업 컨텍스트 정리기 + 설명 초안기"로 확장하는 편이 더 가치가 크다.

## 루트에 먼저 넣어도 좋은 공통 4개

- `new-status`
- `plan-task`
- `update-task`
- `update-troubleshooting`

이 4개는 현재 `infra` 구조를 거의 그대로 가져와도 된다.
다만 구조 문서는 `docs/infrastructure.md`가 아니라 `docs/architecture.md`를 읽도록 바꿔야 한다.

## 루트에서 추가 확장하기 좋은 포인트

### 1. change summary

코드 변경 파일, 최근 diff, 작업 메모를 바탕으로 "무엇이 바뀌었는지"를 짧은 설명 초안으로 만드는 흐름을 추가할 수 있다.

### 2. validation summary

`pnpm check-all`, `uv run make lint`, 테스트 결과를 사람이 읽기 쉬운 검증 요약으로 바꾸는 흐름을 붙일 수 있다.

### 3. README draft

기능 완료나 실험 결과를 루트 README, 포트폴리오 문서, 변경 요약 문서로 변환하는 초안 생성 흐름을 붙일 수 있다.

## 루트용으로 바꾸면 좋은 입력

`infra`처럼 상태 파일과 task만 넣는 것보다 아래 입력을 같이 묶는 편이 좋다.

- 최신 `status`
- 최신 `task`
- 관련 skill 문서
- 최근 수정 파일 목록
- 최근 diff 요약
- 테스트 결과
- 작업 메모 또는 대화 요약

## 추천 적용 순서

1. 공통 4개 워크플로를 루트에 먼저 이식
2. 구조 문서를 `docs/architecture.md` 기준으로 조정
3. `update-task` 입력에 코드 변경 요약이나 작업 메모를 추가
4. 실제 사용성이 확인되면 `change summary`, `validation summary`, `README draft`로 확장

## 결론

현재 단계에서는 루트에 바로 전부 넣기보다,

- 공통 4개 워크플로 먼저 이식
- 입력 품질을 루트에 맞게 강화
- 그 뒤 루트 전용 확장 추가

순서로 가는 것이 가장 안전하다.
