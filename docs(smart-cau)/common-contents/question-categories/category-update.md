# [기획서] Deep Quest AI 면접 질문 생성 알고리즘: 4대 핵심 카테고리

## 1. 개요 및 도입 배경

본 문서는 'Deep Quest' 서비스가 사용자 이력서(STAR 데이터)를 기반으로 생성하는 기술 면접 질문의 **대분류 체계(Category System)**를 재정의한다.

### 1.1. 기존 분류 체계의 한계 (AS-IS)

기존 서비스는 4가지 분류(기술 결정 근거, 구현 심화, 성능 및 확장성 분석, 기술 비교)를 제공했으나, 다음과 같은 구조적 한계가 존재했다.

* **카테고리 간 중복(Non-MECE):** '기술 결정 근거'와 '기술 비교'의 평가 영역이 논리적으로 겹쳐, AI가 생성하는 질문의 의도가 분산되거나 중복됨.
* **검증 로직의 부재:** '구현 심화'라는 포괄적 분류로 인해, 단순한 코드 설명(Description)과 치열한 문제 해결(Troubleshooting) 과정이 명확히 구분되지 않음.
* **STAR 연계성 부족:** 사용자의 경험 서사(STAR) 흐름과 카테고리가 1:1로 매핑되지 않아, 맥락에 맞는 날카로운 꼬리 질문 생성에 한계가 있음.

### 1.2. 개선 방향 (TO-BE)

* **STAR 구조와의 정밀 매핑:** 이력서 파싱 데이터인 **STAR(Situation, Task, Action, Result)**의 각 단계를 검증하기 최적화된 카테고리로 재편한다.
* **명확한 평가 기준(Rubric) 확립:** 모호한 '컬쳐핏'이나 중복되는 '단순 비교'를 배제하고, **설계(Design) → 지식(Knowledge) → 검증(Verification) → 확장(Scalability)**으로 이어지는 개발 프로세스 흐름을 따른다.
* **MECE 원칙 적용:** 4가지 카테고리가 상호 배타적이면서도, 기술 면접의 모든 평가 요소를 누락 없이 포함하도록 설계한다.

## 2. 4대 핵심 카테고리 상세 정의

각 카테고리는 개발자의 경험을 시간 순서와 논리적 깊이에 따라 검증하도록 설계되었다.

### 1. 기술적 의사결정 (Technical Decision Making)

> **"코드를 작성하기 전, 설계 단계에서의 논리와 전략을 검증"**

* **평가 목표:** 개발자가 단순히 기술을 사용한 것을 넘어, 해당 기술을 선택한 **'Why'**를 설명할 수 있는지 확인한다. (기존의 '기술 결정 근거'와 '기술 비교'를 논리적으로 통합)
* **STAR 매핑 로직:** **$Situation(상황) + Task(과제) \rightarrow Action(선택)$**
* **주요 검증 포인트:**
  * **적합성(Suitability):** 프로젝트의 규모, 일정, 팀 역량 등 주어진 환경(**$S$**)에서 최적의 선택(**$A$**)이었는가?
  * **트레이드오프(Trade-off):** 기술 도입으로 얻은 이점과 잃은 기회비용을 명확히 인지하고 있는가?
  * **대안 분석(Alternatives):** 단일 선택지가 아닌, 여러 대안 기술과의 비교 분석 과정을 거쳤는가?

### 2. 기술적 깊이와 원리 (Technical Depth & Principles)

> **"구현에 사용된 도구의 작동 원리와 CS 기초 지식을 검증"**

* **평가 목표:** 이력서에 기재된 기술 스택(Skill)의 내부 동작 원리( **'How & What'** )를 깊이 있게 이해하고 있는지 확인한다. (기존 '구현 심화'를 구체화)
* **STAR 매핑 로직:** **$Action(구현의 심화)$**
* **주요 검증 포인트:**
  * **Under the hood:** 프레임워크나 라이브러리의 추상화된 기능 뒤에서 실제 어떤 일이 일어나는지 설명할 수 있는가?
  * **CS Fundamental:** 구현 코드와 연관된 자료구조, 알고리즘, OS, 네트워크 등 컴퓨터 공학 기초 이론을 갖췄는가?
  * **Best Practice:** 해당 언어/기술이 권장하는 디자인 패턴과 안티 패턴을 구분하여 적용했는가?

### 3. 문제 해결 및 검증 (Problem Solving & Verification)

> **"결과를 도출하는 과정에서의 논리적 해결 능력과 성과의 인과관계를 검증"**

* **평가 목표:** 성과(**$R$**)가 우연의 산물이 아니라, 치열한 문제 해결 과정(**$A$**)과 논리적 검증의 결과임을 **증명(Prove)**할 수 있는지 확인한다. (신설 및 강화된 영역)
* **STAR 매핑 로직:** **$Action(해결) \leftrightarrow Result(성과 검증)$**
* **주요 검증 포인트:**
  * **Root Cause Analysis:** 버그나 이슈 발생 시, 표면적 현상이 아닌 근본 원인을 논리적으로 추적했는가?
  * **Measurement:** 성과(속도 개선, 비용 절감 등)를 측정하기 위한 기준선(Baseline)과 지표(Metric)가 타당한가?
  * **Causality:** "이 조치를 취했기 때문에 결과가 나왔다"는 인과관계를 객관적 데이터로 입증할 수 있는가?

### 4. 확장 가능성 (Scalability)

> **"현재의 결과물을 넘어선 미래의 시야와 아키텍처적 한계를 검증"**

* **평가 목표:** 현재 완료된 프로젝트(**$R$**)를 넘어, 시스템이 성장했을 때(Scale-up)를 대비한 **엔지니어링적 시야(Insight)**를 평가한다. (기존 '성능 및 확장성'의 미래 지향적 관점 강화)
* **STAR 매핑 로직:** **$Result(결과) \rightarrow Future(확장 및 회고)$**
* **주요 검증 포인트:**
  * **Bottleneck Prediction:** 트래픽이나 데이터가 10배, 100배 증가할 때 발생할 수 있는 잠재적 병목 구간을 예측하는가?
  * **System Design:** 단기적 기능 구현을 넘어, 장기적인 유지보수성과 시스템 아키텍처의 유연성을 고려하는가?
  * **Retrospective:** 현재 코드의 한계를 솔직하게 인정하고, 더 나은 개선안(Next Step)을 제안할 수 있는가?

## 3. 카테고리 간의 유기적 흐름 (Workflow)

이 4가지 카테고리는 독립적으로 존재하지 않으며, 하나의 면접 시나리오 안에서 유기적으로 연결된다.

1. [설계] 상황에 맞는 기술을 선택했는가? (1. 의사결정)
   $\downarrow$
2. [구현] 그 기술을 제대로 알고 썼는가? (2. 깊이와 원리)
   $\downarrow$
3. [검증] 발생한 문제를 해결하고 성과를 증명했는가? (3. 문제 해결)
   $\downarrow$
4. **[확장]** 더 큰 시스템에서도 유효한가? (4. 확장 가능성)

## 4. 카테고리별 권장 답변 프레임워크 (Answer Frameworks)

각 카테고리의 질문에 대해 면접자가 답변할 때 권장되는 구조화된 프레임워크를 정의한다. 기존 STAR 구조를 범용적으로 적용하는 대신, **질문 카테고리의 평가 의도에 최적화된 답변 구조**를 제공한다.

### STAR 구조의 한계

STAR(Situation-Task-Action-Result)는 **행동 면접(Behavioral Interview)**을 위해 설계된 프레임워크다. "팀 갈등을 어떻게 해결했나요?", "리더십을 발휘한 경험은?" 같은 소프트 스킬 질문에는 효과적이지만, 기술 면접에 범용으로 적용하면 다음과 같은 문제가 발생한다:

| 문제 | 설명 |
|------|------|
| **평가 의도 불일치** | "Redis 내부 동작 원리를 설명하세요" 같은 질문에 Situation부터 시작하면 핵심을 놓침 |
| **불필요한 서사** | 기술적 깊이를 평가하는 질문에 프로젝트 배경 설명은 시간 낭비 |
| **결론 지연** | 면접관이 원하는 핵심 답변이 마지막(Result)에 나와서 집중력 저하 |
| **카테고리별 차이 무시** | 의사결정, 원리 설명, 문제 해결, 확장성 질문은 각각 다른 구조가 필요 |

따라서 Deep Quest는 **질문 카테고리별 최적화된 프레임워크**를 제공하여, 면접관의 평가 의도에 맞는 답변 훈련을 지원한다.

### 4.0. 두괄식 원칙 (Conclusion First)

> **모든 프레임워크는 "결론 먼저" 원칙을 따른다.**

실제 면접에서 면접관은 먼저 **결론**을 듣고 싶어한다. 장황한 배경 설명 후에 결론을 말하면 핵심을 놓치기 쉽다.

```
✅ 좋은 답변 흐름:
"PostgreSQL을 선택했습니다. (결론)
 주요 이유는 ACID 보장과 팀의 5년 경험이었습니다. (핵심 근거)
 대안으로 MongoDB와 DynamoDB를 검토했는데..." (상세 설명)

❌ 피해야 할 답변 흐름:
"먼저 프로젝트 상황을 설명드리면...
 여러 대안을 검토했는데...
 결론적으로 PostgreSQL을 선택했습니다." (결론이 마지막)
```

### 4.1. 기술적 의사결정 → Trade-off Framework

> **"왜 그 기술을 선택했는가?"**에 대한 체계적인 답변 구조

| 섹션 | 설명 | 예시 |
|------|------|------|
| **Core Answer** | 결론과 핵심 근거를 먼저 | "PostgreSQL을 선택했습니다. ACID 보장과 팀의 5년 경험이 주요 이유였습니다..." |
| **Decision Context** | 대안 비교와 평가 기준 | "MongoDB와 DynamoDB를 검토했고, 데이터 일관성, 팀 역량, 개발 일정을 기준으로..." |
| **Hindsight** | 회고와 교훈 | "돌이켜보면 읽기 복제본을 초기부터 구성했으면 6개월차 성능 이슈를 예방했을 것입니다..." |

### 4.2. 기술적 깊이와 원리 → Layered Explanation

> **"그 기술이 어떻게 동작하는가?"**에 대한 계층적 설명 구조

| 섹션 | 설명 | 예시 |
|------|------|------|
| **Core Answer** | 핵심 개념 정의를 먼저 | "이벤트 소싱은 현재 상태가 아닌 상태 변경 이벤트를 저장하는 패턴입니다. 완전한 감사 추적이 가능합니다..." |
| **How It Works** | 내부 동작 원리 | "이벤트는 불변 로그에 append되고, 현재 상태는 리플레이로 재구성됩니다. 성능을 위해 스냅샷을..." |
| **My Application** | 실제 프로젝트 적용 경험 | "주문 관리 시스템에서 Kafka를 이벤트 스토어로 사용해 고객 분쟁 시 정확한 순서 재현이 가능했습니다..." |

### 4.3. 문제 해결 및 검증 → Root Cause Analysis

> **"그 문제를 어떻게 해결했는가?"**에 대한 논리적 추적 구조

| 섹션 | 설명 | 예시 |
|------|------|------|
| **Core Answer** | 원인과 해결책을 먼저 | "원인은 1천만 행 테이블의 인덱스 누락이었고, 복합 인덱스 추가로 P99가 5초에서 150ms로 개선되었습니다..." |
| **Discovery** | 증상 발견과 조사 과정 | "사용자 체크아웃 5초 지연 신고 후 APM 트레이스로 인벤토리 서비스 지연을 확인했고..." |
| **Validation** | 검증 방법과 재발 방지 | "인덱스 추가 후 P99 150ms 달성을 확인했고, 500ms 초과 알림과 대량 임포트 후 자동 통계 갱신을 설정했습니다..." |

### 4.4. 확장 가능성 → System Thinking

> **"더 큰 규모에서도 동작하는가?"**에 대한 시스템적 사고 구조

| 섹션 | 설명 | 예시 |
|------|------|------|
| **Core Answer** | 핵심 확장 전략을 먼저 | "읽기는 복제본 라우팅, 쓰기는 tenant_id 샤딩으로 100배 성장을 대비할 수 있습니다..." |
| **Analysis** | 현재 상태와 병목 예측 | "현재 10K RPM을 단일 PostgreSQL로 처리 중이고, 100K에서는 연결 풀 고갈과 쓰기 경합이 예상됩니다..." |
| **Trade-offs** | 확장 시 트레이드오프 | "샤딩은 수평 확장을 가능케 하지만 크로스샤드 쿼리가 복잡해지고 일부 데이터는 비정규화가 필요합니다..." |

### 4.5. 프레임워크 적용 원칙

1. **두괄식 (결론 먼저)**: 모든 답변은 Core Answer로 시작하여 결론과 핵심 근거를 먼저 전달
2. **카테고리-프레임워크 1:1 매핑**: 질문의 카테고리에 따라 해당 프레임워크를 사용
3. **간결성**: 3개 섹션으로 구성하여 2-3분 내 전달 가능한 분량 유지
4. **경험 기반**: 모든 섹션에서 본인의 실제 경험을 구체적으로 언급

## 5. 기대 효과 (Value Proposition)

이 분류 체계를 통해 Deep Quest 사용자는 다음과 같은 가치를 얻는다.

* **명확한 자기 객관화:** 기존의 모호했던 피드백에서 벗어나, 자신이 '설계, 구현, 검증, 확장' 중 어느 부분이 취약한지 정확히 파악 가능.
* **구조화된 답변 훈련:** 면접관의 의도(Why, How, Prove, Future)를 파악하고, 이에 맞춰 **카테고리별 답변 프레임워크**로 답변하는 논리적 훈련 가능.
* **고품질 이력서 완성:** 질문에 답하는 과정에서 이력서상의 빈약한 논리(ex. 성과의 근거 부족)를 발견하고 보완할 수 있음.
