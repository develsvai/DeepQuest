# 작업 계획

날짜: `2026-03-27`

## 계획

- 루트 애플리케이션 운영 문서 구조를 `docs/` 기준으로 정리한다
- 루트 workflow를 날짜별 `status`와 `task` 파일 구조로 전환한다
- `web/`, `ai/`, `infra/` 경계를 루트 문서와 skill 기준으로 다시 명확히 맞춘다

## 진행

1. 루트 애플리케이션 운영 문서 구조를 `docs/` 기준으로 정리했다
  - 루트 활성 문서 계층을 `docs(haru2)/status/`, `docs(haru2)/tasks/`, `docs(haru2)/architecture.md`, `docs(haru2)/troubleshooting/`, `docs(haru2)/reference/` 기준으로 맞췄다
  - 문서 생성과 이동 시 참고할 `docs(haru2)/reference/document-writing-rules.md`를 추가했다
   - 트러블슈팅 README와 관련 규칙도 현재 구조에 맞게 정리했다
2. 루트 workflow를 날짜별 `status`와 `task` 파일 구조로 전환하고 역할을 분리했다
   - `new-status` 기준으로 날짜별 상태 파일을 누적하는 구조를 정리했다
   - `plan-task`는 `계획` 초안만 담당하고, `update-task`가 `진행`과 `보류 / 다음 액션`을 갱신하도록 역할을 분리했다
   - `update-status`는 항상 가장 최신 상태 파일을 기준으로 갱신하도록 문서와 skill 규칙을 맞췄다
   - `update-troubleshooting`은 기존 사례를 먼저 확장하고, 기존 범주로 설명되지 않을 때만 새 문서를 만들도록 정리했다
3. `web/`, `ai/`, `infra/` 경계를 루트 문서와 skill 기준으로 다시 명확히 맞췄다
   - 루트는 `architecture`, 인프라는 `infrastructure`를 유지하도록 구조 문서 명칭을 구분했다
   - 루트와 `infra`는 공통 워크플로 기능은 맞추되, `infra`만 `collect-live-state`, `update-infra-state`, `verify-deploy` 같은 운영 전용 skill을 가지도록 정리했다
   - `agent-template`도 현재 루트와 `infra`의 문서 구조, skill 이름, 날짜별 workflow 기준에 맞춰 동기화했다

## 보류 / 다음 액션

1. 남아 있는 옛 경로 참조를 발견 시 계속 정리한다
2. 실제 루트 애플리케이션 구현 작업에서 `new-status -> plan-task -> update-task -> update-status` 흐름을 반복 적용하며 운영성을 검증한다
3. 필요 시 루트 작업용 `reference/` 문서를 더 구조화한다
