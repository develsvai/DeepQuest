# 상태

날짜: `2026-03-28`
확인 시점: `2026-02-08 ~ 2026-03-28`
대상 범위: `web`, `ai`, `애플리케이션 변경 흐름`

이 문서는 기존 루트 상태 파일과 별개로, 과거 문서와 저장소 반영 내용을 바탕으로 `web/`와 `ai/`의 변경 흐름만 따로 정리한 예외 상태 문서다.

## 의도

- 인프라 흐름과 분리해서, `2026-02-08 ~ 2026-03-28` 구간에 확인 가능한 애플리케이션 변경 축을 `web/`와 `ai/` 중심으로 압축 정리한다

## 현재 상태

- `web/` 쪽 변경 흐름은 주로 배포와 운영 연동을 안정화하는 방향에 가까웠고, 프로덕션 URL, 인증, Sentry, Prisma 연결, webhook 연동 같은 런타임 정합성 확인이 중심이었다
- `web/`은 같은 기간 동안 대규모 구조 변경보다는 배포 환경과 맞물린 설정 정리, 외부 연동 정합성, DB 연결 안정화 쪽의 영향이 더 컸다
- `ai/` 쪽은 `langgraph build` 기반 운영 전환 이후 실제 애플리케이션 수정 비중이 더 컸고, LangGraph 그래프를 운영 환경에 맞게 바꾸는 흐름이 누적됐다
- `ai/`에서는 `N_JOBS_PER_WORKER` 적용, session affinity 전제 정리, Redis/Postgres 연동 구조 고정, 주요 그래프 런타임 포트와 실행 경로 조정 같은 운영 결합형 수정이 먼저 축적됐다
- 이후 처리량 개선 구간에서는 `resume_parser`, `question_feedback_gen`, `question_gen`, `jd_structuring`, `jd_to_text` 같은 주요 그래프 경로를 중심으로 async 전환이 진행됐고, `.invoke()` 계열 호출을 `.ainvoke()` 중심으로 옮기며 worker 점유 시간을 줄이는 흐름이 확인됐다
- `resume_parser`는 PDF 다운로드와 재시도 경로를 async 기반으로 바꾸는 축이 있었고, `jd_to_text`는 동기 stream 처리로 런타임을 붙잡는 구간을 완화하는 방향으로 정리됐다
- `question_feedback_gen`, `question_gen`, `jd_structuring`은 공용 Gemini rate limit 연동과 async 호출 구조를 통해, 단순 파드 수 조정만으로 해결되지 않던 AI 처리 병목을 애플리케이션 단에서도 줄이는 방향으로 움직였다
- 정리하면 같은 기간 동안 `web/`은 운영 정합성 보강 비중이 컸고, `ai/`는 실제 처리 경로와 호출 방식까지 손대는 성능/런타임 수정 비중이 더 컸다

## 알려진 이슈

- 이 문서는 애플리케이션 변경 흐름 요약용이라, 개별 커밋 단위의 모든 세부 수정이나 UI 변경 내역을 전부 나열하지는 않는다
- `web/` 변경은 운영 문서와 장애 기록에는 남아 있지만, `ai/`에 비해 흐름 문서가 덜 직접적이라 세부 구현 축은 저장소 코드와 함께 다시 봐야 한다
