# 작업 계획 수립

## 목표

가장 최신 `docs/status/status-YYYY-MM-DD.md`의 `다음 작업`을 중심으로 필요한 코드/문서/설정 조회를 거쳐 번호가 붙은 구체적인 작업 계획으로 바꾼다.
핵심은 `status`의 작업 축을 그대로 이어받아, 같은 날 `task`의 `계획`이 `status`의 `다음 작업`과 같은 내용을 더 구체적으로 보이게 만드는 것이다.

## 작업 순서

1. `docs/status/README.md`와 가장 최신 `status-YYYY-MM-DD.md`를 읽는다
2. `현재 상태`는 지금까지 확인된 사실로 보고 짧게만 파악한다
3. `다음 작업`에 적힌 문장을 현재 이어갈 작업 방향으로 본다
4. 그 작업을 구체화하는 데 필요한 정보가 있으면 관련 문서, 코드, 설정을 먼저 조회한다
5. 조회 결과를 바탕으로 `docs/tasks/task-YYYY-MM-DD.md` 형식의 오늘 작업 파일에 계획을 작성한다

## 규칙

- 작업은 최소 단위로 유지한다
- 명시된 의도 범위를 벗어나지 않는다
- 과도한 설계를 피한다
- 작업 파일은 `docs/tasks/task-YYYY-MM-DD.md` 형식을 따른다
- 기존 날짜 파일은 지우지 않고 새 날짜 파일을 계속 추가한다
- 각 작업 파일에는 최소한 `계획`, `진행`, `보류 / 다음 액션`을 남긴다
- `status`를 읽을 때는 항상 가장 최신 날짜 파일을 기준으로 본다
- task를 생성할 때는 현재 저장소의 task 문서 구조와 헤더 형식을 유지한다
- `현재 상태`는 task의 `계획`에서 길게 반복하지 말고 필요하면 한 줄 수준으로만 반영한다
- `다음 작업`이 task 계획의 주 입력이다
- `계획`은 상태 문서 문장을 복사하지 말고 실제로 수행할 작업 단위로 번호를 붙여 분리한다
- `status`의 `다음 작업`에 있던 큰 작업 축이 `task`의 `계획`에서 사라지거나 다른 축으로 갑자기 바뀌지 않게 한다
- `계획`을 쓰기 전에 필요한 정보가 있으면 먼저 조회하고, 그 조회 결과를 바탕으로 작업을 쪼갠다
- 조회가 필요할 때는 관련 skill과 스크립트를 우선 사용한다
- `계획`은 "무엇을 문서화할지"보다 "무엇을 확인하고, 읽고, 수정할지"가 먼저 드러나야 한다
- `계획` 항목마다 가능하면 조회 대상, 확인 대상, 수정 후보를 짧게 적는다
- 이 skill은 `계획`과 초기 `보류 / 다음 액션`까지만 정리한다
- `진행`은 실제 작업 후 `update-task`에서 갱신한다
- `status`의 `다음 작업`이 이미 과거에 완료된 흐름 요약이라면, `task`는 그 흐름을 더 세밀한 실행 단위로만 풀고 다른 주제로 튀지 않는다
