# 상태 문서 시작

## 목표

새 작업일의 기준 상태 문서를 만든다. 이 문서는 하루 시작 시점의 기준만 담고, 당일 진행 추적은 `task` 문서로 넘긴다.
핵심은 전날 상태를 복사하는 것이 아니라, 전날 `status`의 미완료 작업과 최신 `task`의 실제 진행 결과를 대조해 오늘 시작 기준으로 다시 정리하는 것이다.
또한 날짜가 바뀌어도 작업 축이 갑자기 바뀌지 않도록, 전날 `task`에서 실제로 끝난 일과 남은 일을 기준으로 다음날 문맥이 이어지게 만드는 것이 목적이다.

## 작업 순서

1. `docs/status/README.md`를 읽는다
2. 가장 최신 `docs/status/status-YYYY-MM-DD.md`를 확인한다
3. 가장 최신 `docs/tasks/task-YYYY-MM-DD.md`를 함께 확인한다
4. 오늘 날짜 파일이 없으면 `docs/status/status-YYYY-MM-DD.md`를 새로 만든다
5. 전날 `status`와 가장 최신 `task`를 바탕으로 아래를 오늘 기준으로 정리한다
   - `의도`
   - `현재 상태`
   - `알려진 이슈`
   - `다음 작업`
6. 정리할 때는 아래 매핑 규칙을 따른다
   - `현재 상태`: 전날 `status`의 `다음 작업` 중 최신 `task`의 `진행`에서 실제로 완료되었거나 확인된 내용만 추려 오늘 기준 상태로 다시 쓴다
   - `알려진 이슈`: 전날 `status`의 `알려진 이슈` 중 아직 해결되지 않은 항목과, 최신 `task`의 `진행`에서 새로 드러난 미해결 이슈를 합쳐 정리한다
   - `다음 작업`: 전날 `status`의 `다음 작업` 중 아직 끝나지 않은 항목과 최신 `task`의 `보류 / 다음 액션`을 합쳐 바로 이어서 할 일만 남긴다
7. 전날 `task`의 큰 작업 축이 무엇이었는지 확인하고, 오늘 `status`도 같은 축을 이어받는지 마지막에 점검한다

## 규칙

- `status`는 단일 파일이 아니라 날짜별 로그로 누적한다
- 같은 날짜 파일이 이미 있으면 중복 생성하지 않고 그 파일을 계속 갱신한다
- 다른 skill이 `status`를 읽는다고 하면 가장 최신 날짜 파일을 기준으로 본다
- 확인되지 않은 내용은 새 상태 파일에 사실처럼 적지 않는다
- 새 상태 파일은 현재 `docs/status/` 문서 구조와 기존 상태 문서 톤을 따른다
- 기본 섹션은 `의도`, `현재 상태`, `알려진 이슈`, `다음 작업`을 유지한다
- 전날 `status`의 `현재 상태`를 오늘 파일로 그대로 복사하지 않는다
- 전날 `status`의 `다음 작업`과 최신 `task`의 `진행`을 먼저 대조한 뒤, 완료된 결과만 오늘 `현재 상태`로 승격한다
- 전날 `task`의 `진행`에 없는 사실을 오늘 `현재 상태`에 새로 넣지 않는다
- 최신 `task`의 `계획`은 참고용일 뿐이며, 실제 반영 기준은 `진행`과 `보류 / 다음 액션`이다
- 전날 이슈가 해결되었다면 오늘 `알려진 이슈`에 남기지 않는다
- 새로 생긴 이슈라도 최신 `task`의 `진행`에 근거가 있으면 오늘 `알려진 이슈`에 반영한다
- `현재 상태`는 하루 시작 시점 기준으로 한 번 정리한 뒤, 당일 진행 때문에 다시 덮어쓰지 않는다
- `알려진 이슈`와 `다음 작업`도 전날 `status`와 가장 최신 `task`의 미완료 사항을 기준으로 오늘 시작 시점 관점에서만 정리한다
- 당일 진행 상황, 중단 지점, 실제 실행 기록은 `status`가 아니라 `task`에 남긴다
- 오늘 `status`의 `다음 작업`은 곧이어 작성될 오늘 `task`의 큰 작업 축과 자연스럽게 연결돼야 한다
- 날짜가 바뀔 때 작업 축을 바꾸려면, 전날 `task`의 `진행`이나 `보류 / 다음 액션`에 그 전환 근거가 먼저 있어야 한다
