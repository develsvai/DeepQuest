## Violations by Rule Section

### CRITICAL VIOLATIONS - Design Token Usage (system.md Sections 1-7)

#### 1. Direct Tailwind Classes for Dynamic Sizing

**Rule Reference**: Section 1 - "NEVER Use Hardcoded Colors" & Section 8 - Component Usage Rules
**File Location**: StatusBadge.tsx:67, StatusBadge.tsx:87, StatusBadge.tsx:108
**Current Implementation**:

```tsx
className={`text-${size === 'sm' ? 'xs' : 'sm'} font-medium`}
```

**Required Correction**: Use designTokens for font sizing or static classes

```tsx
className={size === 'sm' ? 'text-xs font-medium' : 'text-sm font-medium'}
```

**Rationale**: Dynamic class generation breaks Tailwind's purging and violates consistent sizing tokens

#### 2. String Concatenation for Background Colors

**Rule Reference**: Section 1 - "NEVER Use Hardcoded Colors"
**File Location**: QuestionCard.tsx:96, FollowUpQuestionItem.tsx:40
**Current Implementation**:

```tsx
backgroundColor: designTokens.colors.muted.DEFAULT + '40',
```

**Required Correction**: Use proper CSS variables with opacity

```tsx
backgroundColor: `${designTokens.colors.muted.DEFAULT}40`, // Or use CSS custom property with opacity
```

**Rationale**: String concatenation may not work reliably with CSS variables

#### 3. Hardcoded Color Values in Utility Functions

**Rule Reference**: Section 1 - "NEVER Use Hardcoded Colors"
**File Location**: mock-data.ts:283-299
**Current Implementation**:

```tsx
case 'DEEP':
  return 'text-green-600 bg-green-50 border-green-200'
```

**Required Correction**: Use designTokens exclusively

```tsx
case 'DEEP':
  return {
    color: designTokens.colors.rating.deep,
    backgroundColor: `${designTokens.colors.rating.deep}10`,
    borderColor: designTokens.colors.rating.deep
  }
```

**Rationale**: These functions bypass the design system and introduce hardcoded colors

#### 4. Raw HTML Elements in Progress Section

**Rule Reference**: Section 8 - "No Raw HTML Elements for UI Components"
**File Location**: ExperienceHeader.tsx:229
**Current Implementation**: Missing Progress component usage (should be present but not found in current implementation)
**Required Correction**: Implement Progress component from design system


**Rationale**: Progress visualization should use design system components
