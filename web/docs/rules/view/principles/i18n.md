# Internationalization (i18n) Rules

> **Claude Code Memory**: This document is summarized in `.claude/rules/frontend/i18n.md` for automatic rule loading.

## Core Principle

All user-facing text MUST use next-intl. No hardcoded strings or conditional locale checks in components.

## NEVER Do This

```tsx
// ❌ Hardcoded text
<span>Submit</span>

// ❌ Conditional locale checks
{locale === 'ko' ? '제출' : 'Submit'}

// ❌ Object literals for translations
const labels = {
  ko: '제출',
  en: 'Submit'
}

// ❌ Inline translations
<Badge>{isCompleted ? (locale === 'ko' ? '완료' : 'Completed') : (locale === 'ko' ? '대기' : 'Pending')}</Badge>
```

## Translation File Structure

### Location

- `/locales/ko/*.json` - Korean translations
- `/locales/en/*.json` - English translations

### Naming Convention

```json
{
  "namespace": {
    "section": {
      "action": "제출",
      "label": "이름",
      "message": "성공",
      "status": {
        "completed": "완료",
        "pending": "대기"
      }
    }
  }
}
```

## ICU Message Format (Required Knowledge)

### Pluralization

Use ICU plural syntax for count-based messages:

```json
{
  "followers": "{count, plural, =0 {팔로워 없음} =1 {팔로워 1명} other {팔로워 #명}}",
  "items": "{count, plural, =0 {No items} one {# item} other {# items}}"
}
```

```tsx
t('followers', { count: 0 }) // "팔로워 없음"
t('followers', { count: 1 }) // "팔로워 1명"
t('followers', { count: 100 }) // "팔로워 100명"
```

### Select (Conditional Text)

Use ICU select for conditional rendering based on values:

```json
{
  "status": "{status, select, online {온라인} offline {오프라인} away {자리비움} other {알 수 없음}}",
  "gender": "{gender, select, female {She} male {He} other {They}} is online."
}
```

```tsx
t('status', { status: 'online' }) // "온라인"
t('gender', { gender: 'female' }) // "She is online."
```

### Date and Number Formatting in Messages

```json
{
  "memberSince": "Member since {date, date, short}",
  "price": "Price: {amount, number, currency}",
  "percentage": "Discount: {value, number, percent}"
}
```

## Implementation Patterns

### 1. Dynamic Values

```json
{
  "followUpQuestions": "꼬리 질문 {count}개",
  "duration": "{years}년 {months}개월"
}
```

```tsx
t('followUpQuestions', { count: 5 }) // "꼬리 질문 5개"
t('duration', { years: 2, months: 3 }) // "2년 3개월"
```

### 2. Rich Text with React Components (`t.rich`)

Returns React nodes - **preferred method** for embedding components:

```json
{
  "terms": "By clicking submit, you agree to our <link>Terms of Service</link>.",
  "important": "This is <bold>very important</bold>."
}
```

```tsx
t.rich('terms', {
  link: chunks => <Link href='/terms'>{chunks}</Link>,
})

t.rich('important', {
  bold: chunks => <strong>{chunks}</strong>,
})
```

### 3. HTML Markup (`t.markup`)

Returns HTML string. **Use with caution** - only for trusted, developer-controlled content:

```json
{
  "markup": "This is <b>important</b>"
}
```

```tsx
// Returns 'This is <b>important</b>'
const html = t.markup('markup', {
  b: chunks => `<b>${chunks}</b>`,
})

// ⚠️ SECURITY: Only use with trusted translation content
// Translation files are developer-controlled, so this is safe
// NEVER use t.markup with user-generated content
```

> **Recommendation**: Prefer `t.rich()` over `t.markup()` whenever possible. Use `t.markup()` only when you specifically need an HTML string (e.g., for email templates or third-party libraries requiring HTML strings).

### 4. Self-Closing Tags

```json
{
  "message": "Hello,<br></br>how are you?"
}
```

```tsx
t.rich('message', {
  br: () => <br />,
})
```

### 5. Conditional Translation Keys (`t.has`)

Check if a translation key exists before using:

```tsx
const t = useTranslations('profile')

// Check before rendering optional content
if (t.has('optionalBio')) {
  return <p>{t('optionalBio')}</p>
}

t.has('title') // true
t.has('unknown') // false
```

### 6. Status/State-Based Text

```tsx
const t = useTranslations('interview-prep.practice.status')
return <span>{t(isCompleted ? 'completed' : 'pending')}</span>
```

## Date, Time, and Number Formatting

### Using `useFormatter`

```tsx
import { useFormatter } from 'next-intl'

function Component() {
  const format = useFormatter()

  return (
    <>
      <p>{format.dateTime(new Date(), 'short')}</p>
      <p>{format.number(1234.56, 'currency')}</p>
      <p>{format.relativeTime(new Date())}</p>
      <p>{format.list(['HTML', 'CSS', 'JS'], 'enumeration')}</p>
    </>
  )
}
```

### Async Server Components

```tsx
import { getFormatter } from 'next-intl/server'

async function ServerComponent() {
  const format = await getFormatter()
  return <p>{format.dateTime(new Date(), 'short')}</p>
}
```

## Namespace Organization

- **Page-level**: `interview-prep.detail`, `interview-prep.practice`
- **Component-level**: `interview-prep.practice.questionCard`
- **Common**: `common.buttons`, `common.status`, `common.messages`

## Error Handling Configuration

Configure in `i18n/request.ts`:

```typescript
import { getRequestConfig } from 'next-intl/server'
import { IntlErrorCode } from 'next-intl'

export default getRequestConfig(async () => {
  return {
    locale,
    messages,

    onError(error) {
      if (error.code === IntlErrorCode.MISSING_MESSAGE) {
        // Expected during development - log only
        console.error(error)
      } else {
        // Bug in the app - report to Sentry
        reportToErrorTracking(error)
      }
    },

    getMessageFallback({ namespace, key, error }) {
      const path = [namespace, key].filter(Boolean).join('.')

      if (error.code === IntlErrorCode.MISSING_MESSAGE) {
        return `[Missing: ${path}]`
      }
      return `[Error: ${path}]`
    },
  }
})
```

## TypeScript Integration

### Type-Safe Translations (next-intl 4.0+)

```typescript
// types/i18n.d.ts
import messages from '@/locales/en/common.json'
import { routing } from '@/i18n/routing'

declare module 'next-intl' {
  interface AppConfig {
    Messages: typeof messages
    Locale: (typeof routing.locales)[number]
  }
}
```

### Enable JSON Type Declarations

```json
// tsconfig.json
{
  "compilerOptions": {
    "allowArbitraryExtensions": true
  }
}
```

### Usage with Types

```tsx
// Autocomplete and type checking
const t = useTranslations('interview-prep.practice')
t('questionList.completed') // ✅ Type-safe
t('invalid.key')            // ❌ TypeScript error

// Locale type is also enforced
import { Link } from '@/i18n/routing'
<Link href="/" locale="en" />  // ✅ Valid
<Link href="/" locale="xx" />  // ❌ TypeScript error
```

## Migration Checklist

When updating components:

1. **Identify all hardcoded text** using:

   ```bash
   grep -r "locale === 'ko'" ./src
   grep -r "['\"]\(완료\|미완료\|제출\)" ./src
   ```

2. **Add translation keys** to `/locales/{locale}/*.json`
3. **Update component** to use `useTranslations` or `getTranslations`
4. **Remove locale prop** if only used for translations
5. **Test both languages** to ensure keys exist

## Testing

Always test with both locales:

```bash
# Korean
http://localhost:3000/ko/your-page

# English
http://localhost:3000/en/your-page
```

## Performance

- Translation keys are loaded per page via `i18n/request.ts`
- Use namespaces to avoid loading unnecessary translations
- Common translations are always loaded via `common.json`
- Use `pick()` to limit messages sent to client components

## Exceptions

The ONLY acceptable hardcoded text:

- Console.log messages (development only)
- Error messages for developers (not user-facing)
- HTML meta tags handled by next-intl separately
